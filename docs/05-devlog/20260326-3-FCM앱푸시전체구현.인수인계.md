## 3. 핵심 변경 내용
1. **DB 스키마 구성**: 정보통신망법 대응 수신 동의 컬럼, 시스템 오류 대응 이력(Log) 테이블 구축 완료
2. **2-Phase Token 인증 동기화**: `usePushNotifications` 내에서 비회원일 시 Token을 LocalStorage/App State에 들고 있다가, 로그인을 완료하면 서버에 Upsert 하도록 생명주기 완벽 동기화 (토큰 증발 이슈 제로화)
3. **토큰 누수 원천 차단 로직**: `auth.ts` 로그아웃 API, `profile.ts`의 탈퇴 API와 연계하여 로그아웃 및 탈퇴시 기기의 Token 활성 상태를 false로 바꿔 좀비 알림 방어.
4. **FCM 발송 엔진 고도화**: Error-Isolated(개별 사용자 전송 실패 시, 타깃 건에만 토큰 비활성화 부여), Topic(전체 메세지), 청크 전송 보완 완료.
5. **관리자 제어반 완성**: 푸시 로그 확인 테이블 구현, All_users/Specific 대상 수동 전송 기능 UI 구축 완료
6. **Background Cron 기능 도입**: Server Entry 파일(`index.ts`)에 매일 24시간 주기로 과거 삭제된 / 미사용 FCM 토큰을 정리하는 `setInterval` 코드 추가.

## 4. 왜 변경했는지 (의사결정 이유)
- 사용자의 모바일 환경(App)으로 넘어가는 플랫폼 서비스에서 브라우저 미접속 상태일 때도 Engagement 창출을 높일 수단이 전무하였으므로, FCM 네이티브 Push 연동이 필수적이었습니다.
- 기기의 고유 Token과 JWT 시스템의 LifeCycle이 동기화 없이 각각 돌 경우 보안 문제 또는 알림 누설 이슈가 발생하므로, "2-Phase Token Register"라는 강한 동기화 패턴을 설계 적용했습니다.
- 백엔드에 대용량 부하 없이 전송하기 위하여 `Promise.allSettled`를 사용해 에러를 무시하면서도, 전파에 실패한 타깃(좀비 토큰 기종)은 DB에서 즉시 `isActive=false` 처리하여 Firebase 비용 및 자원 낭비를 줄이게 강제했습니다.

## 6. 주의사항 / 알려진 이슈
- **개발환경(PC)과 모바일 팝업**: 현재 FCM 시스템은 Capacitor App Wrapper에 의존하므로 Vite Web View(개발자 도구 등) 위에서는 API 토큰 Upsert 콘솔까지만 뜹니다.
- **`Action_Url` 딥링크 규칙**: 클라이언트 앱의 React-Wouter 라우팅 기준 형식을 따릅니다. 관리자 패널에서 수동 발송 시 `/mymissions`, `/admin` 과 같은 형태로 보내야 합니다.
- **Firebase Google Services**: 기존 적용되어 있는 `firebase-admin.ts` 의 서비스 워커 권한을 그대로 가져와 쓰므로, 프로젝트 ID나 서버 키를 새로 교체했을 땐 반드시 환경 변수(.env) 업데이트가 필요합니다.

## 7. 연관 파일
- 앱의 푸시 수신 동작은 오직 `client/src/App.tsx` 최상단의 `CapacitorGlobalListeners`에서 바인딩됩니다. 추가적인 Native Listener가 필요할 땐 반드시 해당 선언부를 확인하세요.
- `server/services/notifications.ts`가 핵심 병합 통로이므로, 서버 코드 작업 중 알림을 트리거하려면 항상 해당 파일의 `createNotification` 헬퍼 함수를 통해 통일성 있게 호출하시길 강력히 권장합니다.
