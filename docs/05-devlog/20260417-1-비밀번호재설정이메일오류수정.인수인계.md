작성일: 2026-04-17

## 2. 핵심 변경 내용 및 원인(왜 변경했는지)

**[Bug 1] 프로덕션 환경의 비밀번호 찾기 이메일 URL 오류**
- **기존:** `PRODUCTION_DOMAIN` 환경변수 부재 시 프로덕션 환경이어도 하드코딩된 `http://localhost:5000` 도메인으로 이메일에 링크가 생성되는 버그가 존재했습니다.
- **수정:** 링크 생성 시 `req.protocol` 및 `req.headers.host`를 읽어들여 런타임에 동적으로 실제 호스트 URL을 삽입하도록 변경했습니다. 

**[Bug 2] 발급된 재설정 토큰이 실시간으로 만료되는 버그 ("유효하지 않은 링크")**
- **기존:** JS의 `new Date()` 객체를 Drizzle ORM을 통해 `timestamp without time zone` 컬럼에 바로 삽입하고 조회하면서 `pg` 드라이버의 캐스팅 정책에 의해 시스템이 9시간의 타임존 오차를 발생시켰습니다. 생성된 토큰이 백엔드 내부적으로 8시간 전에 이미 만료된 것으로 처리되어 이메일 링크를 클릭하자마자 400 에러를 반환했습니다.
- **수정:** 자바스크립트의 시간 객체 생성을 폐기하고, 토큰 생성 및 검증 코드를 PostgreSQL 네이티브인 `sql\`NOW() + interval '1 hour'\``와 `sql\`NOW()\``로 대체하였습니다. 이를 통해 서버 환경과 상관없이 완벽하게 데이터베이스 시간과 동기화되어 유효 기간 로직이 정상 작동합니다.

**[Bug 3] Railway 프로덕션 서버에서 이메일 발송 자체 실패 (ETIMEDOUT)**
- **근본 원인:** Railway 클라우드 서버는 스팸 발송 악용 방지를 위해 **SMTP 포트(25, 465, 587)로 나가는 모든 아웃바운드 연결을 차단**하고 있었습니다. 이로 인해 Nodemailer가 구글 SMTP 서버(`smtp.gmail.com`)에 접속을 시도하면 10초~2분간 대기 후 `ETIMEDOUT` 에러가 발생하며 메일 발송이 완전히 실패했습니다.
- **해결:** 이메일 발송 엔진을 **Nodemailer(SMTP 방식)에서 Resend(HTTP API 방식)으로 전면 교체**했습니다. Resend는 HTTPS(443포트)를 사용하므로 Railway의 SMTP 차단에 영향받지 않습니다.
- **변경 상세:**
  - `server/services/email.ts` — Nodemailer 코드 전체를 Resend SDK(`resend` 패키지)로 교체
  - 기존 HTML 이메일 템플릿과 함수 시그니처(`sendPasswordResetEmail`, `sendPasswordResetSuccessEmail`, `sendVerificationEmail`, `isValidEmail`)는 100% 동일하게 유지하여 호환성 보장
  - 환경변수 `RESEND_API_KEY` 추가 (로컬 `.env` + Railway Variables)

**[추가] 네이버/다음 메일 수신을 위한 DNS 보안 설정**
- 가비아 DNS에 SPF 레코드 추가: `v=spf1 include:_spf.google.com ~all`
- Resend 도메인 인증을 위해 추가 DNS 레코드 4건 등록:
  - TXT `resend._domainkey` (DKIM)
  - MX `send` → `feedback-smtp.ap-northeast-1.amazonses.com.` (우선순위 10)
  - TXT `send` → `v=spf1 include:amazonses.com ~all`
  - TXT `_dmarc` → `v=DMARC1; p=none;`

## 4. 주의사항 / 알려진 이슈
- `/api/auth/forgot-password` 엔드포인트는 보안상 가입되지 않은 이메일 주소를 입력해도, 해커의 계정 추론 방지를 위해 클라이언트에는 `success: true`를 떨어뜨립니다. (메일함에 도착하지 않음) 
운영 모드에서 유저가 "비밀번호 찾기 메일이 왜 안 오나요?"라고 문의할 때, 철자가 틀리거나 가입되지 않은 이메일인지 먼저 확인하도록 안내 메뉴얼에 추가하는 것이 좋습니다.
- **Railway 환경 특성:** Railway는 SMTP 포트를 기본 차단하므로, 향후 이메일 관련 기능(알림, 마케팅 등)을 추가할 때도 반드시 HTTP API 기반 서비스(Resend 등)를 사용해야 합니다. Nodemailer로 되돌리면 프로덕션에서 다시 실패합니다.
- **Resend 무료 플랜 제한:** 일 100통, 월 3,000통 발송 가능. 비밀번호 재설정 용도로는 충분하지만, 대량 마케팅 메일을 보내려면 유료 플랜 업그레이드 필요.
- **환경변수:** `RESEND_API_KEY`가 Railway Variables에 반드시 등록되어 있어야 합니다. 누락 시 이메일 발송 실패.
