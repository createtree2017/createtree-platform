# 세부미션 표시 순서 수정

작성일: 2026-05-11

## 1. 작업 개요

관리자 페이지의 `미션 시스템 관리 > 주제 미션 > 세부미션 관리`에서 드래그앤드롭으로 세부미션 순서를 바꾸면 `잘못된 요청 형식입니다` 오류가 발생하던 문제를 수정했다.
또한 세부미션 수정 모달에 `표시 순서` 숫자 입력을 추가해 드래그앤드롭 외에도 운영자가 직접 정렬 값을 조정할 수 있도록 했다.

## 4. 핵심 변경 내용

- 세부미션 드래그 완료 시 `subMissionOrders: [{ id, order }]` 형식으로 저장 요청을 보내도록 수정했다.
- 서버의 세부미션 순서 변경 API가 `subMissionOrders`를 검증해 처리하도록 유지하면서, 기존 `subMissionIds` 배열 요청도 `order` 값으로 변환해 처리할 수 있게 방어적으로 보강했다.
- 세부미션 순서 업데이트를 트랜잭션으로 묶어 일부만 업데이트되는 위험을 줄였다.
- 세부미션 목록 조회 정렬을 `order ASC, id ASC`로 보강해 같은 순서 값이 있어도 표시 순서가 흔들리지 않게 했다.
- 세부미션 생성/수정 컨트롤러가 검증된 데이터를 서비스에 전달하도록 고쳤다.
- 세부미션 부분 업데이트 시 전달되지 않은 `startDate`, `endDate`, `sequentialLevel`이 임의로 초기화되지 않도록 서비스 정규화 로직을 수정했다.
- 세부미션 수정 모달에 `표시 순서` 숫자 입력을 추가하고, `순차 등급`과 별개의 정렬 값임을 설명 문구로 명시했다.

## 5. 변경 이유

오류의 직접 원인은 프론트와 서버의 API 요청 계약 불일치였다. 프론트는 ID 배열을 보내고 있었지만 서버는 ID와 순서가 포함된 객체 배열을 기대하고 있어 Zod 검증에서 400 오류가 발생했다.

또한 코드 검토 중 서버 컨트롤러가 검증한 데이터를 사용하지 않고 원본 요청 본문을 서비스로 넘기는 구조와, 부분 업데이트가 날짜/순차등급을 의도치 않게 초기화할 수 있는 구조가 확인되었다. 순서 기능을 고치는 김에 같은 세부미션 저장 경로의 데이터 안정성을 함께 보강했다.

## 6. 동작 확인 상태

- `npm run build` 성공
- `npm run check` 실패: 기존 `PersonaCategoryManagement.tsx`, `PersonaManagement.tsx`, `server/services/milestones.ts` 타입 오류로 실패했다. 이번 변경 파일 관련 신규 빌드 오류는 확인되지 않았다.
- 브라우저 검증은 사용자 요청에 `!!테스트!!`가 포함되지 않아 자동 수행하지 않았다.

## 7. 주의사항 / 알려진 이슈

- `표시 순서(order)`는 목록 정렬용 값이고, `순차 등급(sequentialLevel)`은 이전 등급 완료 후 다음 등급을 여는 잠금 기능이다. 두 값은 의도적으로 분리했다.
- 동일한 표시 순서 값이 여러 개일 수 있다. 이 경우 `id` 오름차순으로 안정 정렬된다.
- 드래그앤드롭과 숫자 입력이 같은 `order` 값을 수정하므로, 둘 중 마지막으로 저장한 방식이 최종 순서가 된다.

## 8. 수동 테스트 참고

1. `localhost:5000/api/dev/auto-login` 접속
2. 관리자 페이지 → 메뉴관리 → 문화센터 → 미션 시스템 관리 → 주제 미션 이동
3. 특정 주제 미션의 세부미션 관리 열기
4. 드래그 핸들로 세부미션 순서를 변경 → 오류 토스트 없이 목록이 갱신되는지 확인
5. 세부미션 수정 모달 열기 → `표시 순서` 값을 변경해 저장
6. 목록 순서가 변경되고 `순차 등급` 값은 별도로 유지되는지 확인
