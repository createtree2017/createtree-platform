# Android 갤러리 다운로드 저장 인수인계

작성일: 2026-05-02

## 1. 배경

Android Capacitor 앱에서 갤러리 이미지 다운로드 버튼을 눌러도 파일이 저장되지 않았다.
기존 구현은 웹 브라우저용 `a.download` 방식에 의존하고 있었고, Android WebView에는 해당 다운로드를 실제 갤러리 저장으로 연결하는 네이티브 처리가 없었다.

## 2. 변경 내용

- `client/src/components/GalleryEmbedSimple.tsx`
  - 다운로드 시 Android 앱 네이티브 브리지가 있으면 우선 사용하도록 분기했다.
  - 기존 PC Chrome, 모바일 Chrome, iOS PWA 다운로드 흐름은 브리지가 없을 때 그대로 유지한다.
- `client/src/utils/android-gallery-download.ts`
  - `/api/download-image/:id`를 인증 쿠키 포함 `fetch`로 받아 Blob/Base64로 변환한다.
  - Android 네이티브 브리지(`window.CreateTreeAndroid.saveImageToGallery`)로 이미지 데이터와 파일명을 전달한다.
- `android/app/src/main/java/com/createtree/app/MainActivity.java`
  - Capacitor WebView에 `CreateTreeAndroid` JavaScript interface를 등록했다.
- `android/app/src/main/java/com/createtree/app/AndroidGalleryBridge.java`
  - 전달받은 Base64 이미지 데이터를 `MediaStore.Images`에 저장한다.
  - 저장 위치는 `Pictures/AI문화센터`이며, Android 갤러리/사진 앱에 바로 노출되는 방식이다.
- `android/app/build.gradle`
  - Play Console 새 릴리스 업로드를 위해 `versionCode 9`, `versionName "1.9"`로 갱신했다.

## 3. 왜 이렇게 변경했는가

- `DownloadManager`로 저장하면 일반적으로 다운로드 폴더 중심이라 사용자가 갤러리에서 바로 찾기 어렵다.
- 앱 요구사항은 “갤러리에서 바로 보이는 것”이므로 Android 공식 미디어 저장소인 `MediaStore.Images`를 사용했다.
- 서버 API는 이미 인증/소유권 확인과 GCS Signed URL 재발급을 처리하므로, 프론트에서 같은 API를 `fetch`한 뒤 네이티브 저장만 담당하게 해 기존 서버 로직을 재사용했다.

## 4. 검증

- `npx tsc --noEmit --skipLibCheck --target ES2022 --lib es2022 --lib dom --moduleResolution bundler --module ESNext client/src/utils/android-gallery-download.ts`
  - 통과.
- `npm run build`
  - 통과.
  - `android-gallery-download-*.js` 청크 생성 확인.
- `npx tsc --noEmit`
  - 실패.
  - 이번 변경과 무관한 기존 전역 TypeScript 오류 다수 존재.
- `android/gradlew.bat :app:compileDebugJavaWithJavac`
  - 실패.
  - 현재 환경 Java가 17인데 Android Gradle 설정이 Java 21(`JavaVersion.VERSION_21`)을 요구함.

## 5. 주의사항

- Android 실기기 검증은 JDK 21 환경에서 APK/AAB 빌드 후 진행해야 한다.
- 앱은 `capacitor.config.ts`에서 Railway 프로덕션 서버를 로드하므로, 웹 배포와 앱 빌드 반영 시점을 함께 확인해야 한다.
- `!!푸시!!` 지시가 없었으므로 git add/commit/push는 수행하지 않았다.
