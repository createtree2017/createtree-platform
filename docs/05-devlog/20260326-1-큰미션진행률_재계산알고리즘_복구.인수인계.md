# [20260326-1] 큰미션 토픽 진행률 연동 버그(로직 유실) 복구

## 1. 작업 개요
- **배경**: 문화센터 미션을 사용자가 완료/승인받았을 때, 연결된 "큰미션"의 성취도(`completedTopics`)가 오르지 않는 치명적인 버그 발생.
- **원인 파악**: 과거 모놀리식 서버 구조(mission-routes.ts 등)를 관심사별로 분리(Layered Architecture)하는 대규모 리팩토링 과정에서 `userBigMissionProgress` 테이블을 업데이트하는 부수효과(Side-Effect) 로직이 타 도메인으로 분류되어 소실된 것으로 확인됨.
- **핵심 목표**: 단순 카운트 방식이 아닌, 사용자가 완료한 전체 서브미션의 카테고리 풀(Set) 데이터를 기반으로 큰미션 달성 여부를 동적으로 재평가하는 안전한 알고리즘으로 복구.

## 2. 변경 파일 및 핵심 변경 내용

- **`server/services/mission/admin.mission.review.service.ts`**
  - **왜 변경했는지**: 관리자가 유저의 일반 미션을 "승인" 또는 "반려"할 때 그 이벤트에 반응하여 큰미션 진척도를 업데이트해야 하므로 `AdminMissionReviewService` 에 코드 삽입.
  - **추가 로직**: `recalculateUserBigMissionProgress(userId)` 프라이빗 메서드 신규 작성.
    - 해당 유저의 모든 *승인된(APPROVED)* 서브미션을 긁어와서 연결된 `categoryId`를 중복 제거(Set)하여 추출.
    - 활성화된 큰미션과 그 하위 토픽들을 순회하며, 유저의 완료 카테고리를 대조해 `completedTopics` 갯수와 `status` 산출.
    - `userBigMissionProgress` 테이블을 `INSERT` 혹은 `UPDATE`.
  - **Hook 연결**: `approveSubmission`, `rejectSubmission`, `bulkUpdateStatus` 메서드 응답 전 단계에 해당 재계산 로직을 비동기로 호출(`.catch` 처리로 메인 로직 속도 저하 방지).

## 4. 주의사항 / 알려진 이슈
- **의존성(부수효과) 주의**: 이 알고리즘은 사용자의 기존 제출물(APPROVED 상태인 전체 subMissionSubmission)을 로드하여 계산합니다. 차후 유저 수가 기하급수적으로 늘 경우 쿼리 최적화가 필요할 수 있으나 현재 규모에서는 가장 안전한 방법입니다.
- **새 파일/라우트/스키마/서비스 수정 시:** 규칙에 따라 수정 직후 `npm run dev` 서버를 반드시 재시작해야 새로운 서비스 인스턴스가 올라가며 정상 작동합니다.
