## 핵심 변경 내용

### API HTML fallback 차단

개발 서버의 Vite fallback에서 미등록 `/api` 또는 `/api/*` 요청은 더 이상 HTML `200 OK`를 반환하지 않고 JSON `404`를 반환한다. 일반 클라이언트 라우트는 기존처럼 HTML fallback을 유지한다.

### 프론트 API 응답 방어

공통 `apiRequest`에서 `/api/*` 요청이 `text/html` 응답을 받으면 성공으로 처리하지 않고 오류를 throw한다. 서버 재시작 누락이나 라우트 미등록 시 성공 토스트가 뜨는 false success를 막기 위한 방어다.

### verify 환경 로딩 보강

`scripts/verify-env.ts`가 `dotenv/config`를 로드하도록 수정했다. 이제 `npm run verify`가 실제 `.env` 기준으로 `DATABASE_URL`, `SESSION_SECRET`, `JWT_SECRET`을 검증한다.

### 기존 TypeScript 오류 정리

`PersonaCategoryManagement.tsx`, `PersonaManagement.tsx`의 TanStack Query 응답 타입을 명시하고, DB nullable/default/JSON 필드를 폼 값으로 안전하게 정규화했다.

`server/services/milestones.ts`는 스키마 기준에 맞춰 `milestoneApplications.milestoneId`를 문자열 식별자 기준으로 정리했다. 카테고리별 통계 계산도 배열 반환값을 객체처럼 인덱싱하던 부분을 필터 기반 계산으로 수정했다.

## 변경 이유

승인취소 API가 서버 재시작 전에는 등록되지 않아 Vite HTML fallback이 `200 OK`로 반환되고, 프론트가 이를 성공으로 오인했다. 같은 유형은 다른 API mutation에서도 재발할 수 있으므로 서버와 클라이언트 양쪽에 방어선을 추가했다.

또한 전체 타입 검증과 `npm run verify`가 실패하는 상태는 배포 전 신뢰도를 낮추므로, 사이트 구동 계약을 바꾸지 않는 범위에서 기존 오류를 함께 정리했다.

## 동작 확인 상태

- `npm run check -- --pretty false`: 통과
- `npm run verify`: 통과
- `git diff --check`: 통과
- `npm run build`: 통과
- 임시 dev 서버 `PORT=5010` 확인:
  - `POST /api/__missing_route_false_success_probe` → `404 application/json`
  - `POST /api/admin/review/submissions/845/cancel-approval` 인증 없음 → `401 application/json`
  - `GET /definitely-client-route` → `200 text/html`

## 주의사항 / 알려진 이슈

- 현재 실행 중인 기존 `localhost:5000` dev 서버에는 서버 파일 변경이 자동 반영되지 않을 수 있다. `server/vite.ts` 변경 확인을 위해서는 `npm run dev` 재시작이 필요하다.
- `npm run build`는 통과하지만 기존과 동일하게 큰 chunk 및 dynamic/static import 혼용 경고가 남아 있다. 이번 작업 범위의 런타임 오류는 아니다.
- git add/commit/push는 `!!푸시!!`가 없어서 수행하지 않았다.
