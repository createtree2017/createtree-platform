# 관리자 회원수정 범위 정리

작성일: 2026-05-14

## 1. 작업 개요

관리자 회원관리에서 회원 정보 수정 시 서버 500 오류가 발생하고 사용자에게 `다시 시도해주세요` 형태의 모호한 메시지가 표시되던 문제를 정리했다. 수정 가능 범위를 명확히 하고, 프론트와 서버가 같은 정책을 따르도록 맞췄다.

## 4. 핵심 변경 내용

- 관리자 회원 수정 API에서 `username`, `email`, `birthdate`를 수정 불가 필드로 명시하고, 전달되면 400 오류와 안내 메시지를 반환하도록 변경했다.
- 관리자 회원 수정 API가 실제로 업데이트하는 필드를 `memberType`, `hospitalId`, `phoneNumber`로 제한했다.
- 전화번호는 서버에서 숫자만 추출해 저장하며, 값이 있을 때는 10~11자리만 허용한다.
- 회원 수정 모달에서 이름, 회원명, 이메일, 생년월일은 읽기 전용 정보로 표시하고 입력 필드에서 제거했다.
- 회원 수정 실패 토스트가 서버의 실제 오류 메시지를 보여주도록 변경했다.

## 5. 왜 변경했는지

기존 UI는 생년월일 같은 수정 불가 성격의 값을 함께 전송했고, 서버는 해당 값을 그대로 업데이트 대상으로 넣을 수 있었다. DB 타입과 맞지 않는 값이 들어오면 500 오류가 발생하고, 운영자는 실제 원인을 알 수 없었다. 따라서 프론트에서는 보내지 않고, 서버에서는 받아도 명확히 거절하는 방식으로 방어했다.

## 6. 동작 확인 상태

- `npm run build`: 성공
- `npx tsc --pretty false --noEmit` 변경 파일명 필터: 관련 신규 오류 없음
- 전체 `tsc`: 기존 `PersonaCategoryManagement.tsx` 등 타입 오류가 남아 있어 전체 통과는 아님
- `git diff --check`: 통과
- `!!테스트!!` 키워드가 없어 브라우저 에이전트 검증은 수행하지 않았다.

## 7. 수동 확인 포인트

- 슈퍼관리자로 관리자 회원관리 진입
- 회원 수정 모달에서 이름/회원명/이메일/생년월일이 읽기 전용으로 보이는지 확인
- 전화번호를 `010-5507-1460`처럼 입력 후 저장했을 때 목록에 하이픈 포함으로 표시되는지 확인
- 개발자도구 Network에서 요청 payload가 `memberType`, `hospitalId`, `phoneNumber` 중심으로 나가는지 확인

## 8. Skill Impact Check

- 관리자 회원 수정 정책과 전화번호 정규화 흐름을 정리했지만, 프로젝트 운영 스킬의 자동 호출 기준이나 배포/DB 정책 변경은 아니다.
- 스킬 문서 업데이트는 불필요하다고 판단했다.
