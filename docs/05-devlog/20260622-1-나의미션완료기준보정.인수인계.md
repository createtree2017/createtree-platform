## 핵심 변경 내용

### 큰미션 완료 계산 공통화

- `server/services/mission/big-mission-progress.service.ts`를 추가했다.
- 사용자별 완료 카테고리는 아래 기준으로만 인정한다.
  - 같은 `categoryId`의 `themeMission`
  - `isActive=true`인 세부미션 1개 이상
  - 해당 활성 세부미션 전체에 대한 사용자 `approved` 제출 존재
- `themeMission.isActive`는 완료 후보에서 강제하지 않았다.
  - 월별/회차별 프로그램이 비활성 처리되어도 과거 완료 이력을 인정할 수 있어야 하기 때문이다.
  - 단, 세부미션은 현재 활성 상태인 항목만 분모/분자로 사용한다.

### 사용자 API 보정

- `GET /api/big-missions`
- `GET /api/big-missions/:id`
- `POST /api/big-missions/:id/apply-reward`

위 세 API가 공통 완료 계산 서비스를 사용하도록 변경했다. 보상 신청 전에도 새 기준으로 100% 완료 여부를 다시 계산한다. `rewardStatus=cancelled` 이력이 있어도 새 기준으로 완료 상태이면 재신청할 수 있고, `approved` 지급완료는 계속 수정/재신청을 차단한다.

### 관리자 재계산 보정

- `admin.mission.review.service.ts`의 기존 큰미션 재계산 중복 로직을 제거하고 공통 서비스 호출로 교체했다.
- 관리자 승인/반려/일괄승인 후 저장 progress도 새 기준으로 갱신된다.
- 보상 신청 상태는 자동 삭제/취소하지 않는다.

### 문화센터 진행률 보정

- 사용자 문화센터 목록/상세/히스토리/나의미션 관련 진행률 계산에서 approved count 조건에 활성 세부미션 필터를 추가했다.
- 문화센터 프로그램 완료 처리에서도 활성 세부미션 approved만 인정하고, 활성 세부미션이 0개면 완료 처리하지 않는다.
- 신청취소, 출석인증, 자동승인 같은 관리자 검수 외 경로에서도 큰미션 저장 progress가 뒤처지지 않도록 재계산 호출을 연결했다.

## 변경 이유

나의미션은 특정 문화센터 프로그램을 직접 들고 있지 않고 카테고리 슬롯으로 운영된다. 사용자가 같은 카테고리의 어떤 월/회차 프로그램을 완료하든 나의미션을 완료할 수 있어야 하지만, 프로그램 하나를 끝까지 완료했다는 기준은 신청 완료가 아니라 관리자가 활성화한 세부미션 전체 승인이어야 한다.

기존 로직은 카테고리 단위 approved 제출 1건만 보고 완료 처리했기 때문에, 신청 승인만으로 완료 선물 신청까지 열리는 문제가 있었다. 이번 작업은 완료 계산 기준을 공통 서비스로 고정해 사용자 화면, 보상 신청, 관리자 검수 후 저장 progress가 서로 다른 기준을 쓰지 않도록 한 것이다.

## 주의사항 / 알려진 이슈

- `db/migrations/20260622_add_reward_cancel_fields.sql`은 파일로 남겼고, 현재 연결된 개발 DB에는 `ADD COLUMN IF NOT EXISTS` 방식으로 적용 및 확인했다. 운영/다른 환경은 배포 마이그레이션 절차에서 별도 적용 여부를 확인해야 한다.
- 이번 작업은 기존 데이터 실제 보정을 수행하지 않았다. 운영 데이터 변경은 dry-run 결과 확인 후 별도 승인으로 진행해야 한다.
- `approved` 지급완료 보상 신청은 자동취소하지 않는다. dry-run에서 수동 검토 대상으로 분리해서 판단해야 한다.
- `rewardStatus=cancelled`는 기존 varchar 컬럼에 새 상태값을 쓰는 방식이다. 관련 UI/API는 반영했지만, 외부 리포트가 이 상태를 별도로 필터링한다면 후속 확인이 필요하다.
- `!!테스트!!` 요청이 아니어서 브라우저 자동 검증은 수행하지 않았다.
- 작업 전부터 존재하던 미추적 인수인계 파일 `docs/05-devlog/20260612-모바일크롬PWA앱홈스크롤불가수정.인수인계.md`는 수정하지 않았다.

## 수동 테스트 순서

1. 운영/다른 환경 DB에 `20260622_add_reward_cancel_fields.sql` 마이그레이션이 적용되어 있는지 확인한다.
2. `localhost:5000/api/dev/auto-login` 접속 후 관리자 계정으로 로그인한다.
3. 관리자 > 문화센터에서 `신청` 세부미션만 승인된 사용자를 준비한다.
4. 사용자 `/mymissions`와 `/mymissions/{id}`에서 해당 진행미션이 완료로 표시되지 않는지 확인한다.
5. 같은 카테고리의 문화센터 프로그램에서 활성 세부미션 전체를 approved로 만든다.
6. 사용자 나의미션 진행미션이 완료되고 완료 선물 신청 폼이 열리는지 확인한다.
7. 관리자 > 나의미션 > 보상 신청 관리에서 pending 신청을 운영취소하고 `운영취소` 필터/사유 표시를 확인한다.

## 2026-06-22 추가 보정 / 세부미션 비활성 완료 조건 동기화

### 확인된 원인

- 나의미션 큰미션 완료 계산 자체는 `bigMissionProgressService.getCompletedCategoryMapForUser`에서 이미 활성 세부미션만 분모로 사용하고 있었다.

  - 분자: 해당 활성 세부미션의 `subMissionSubmissions.status=approved`
  - 비활성 세부미션은 완료 조건에서 제외
- 문화센터 사용자 진행률 계산도 조회 화면 기준으로 활성 세부미션만 분모/분자로 사용하고 있었다.
