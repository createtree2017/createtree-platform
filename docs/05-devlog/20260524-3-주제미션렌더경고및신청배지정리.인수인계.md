# 주제미션 렌더 경고 및 신청 배지 정리

작성일: 2026-05-24

## 1. 작업 개요

관리자 문화센터 `주제 미션` 탭 진입 시 반복되던 `Maximum update depth exceeded` 콘솔 경고를 폴더 목록 상태 동기화 경로에서 차단했다. 또한 세부미션 목록에서 기존 `신청 + requireReview=true` 데이터가 `검수 필요`로 보이는 혼동을 줄이기 위해 실제 신청 정책을 나타내는 배지로 표시했다.

## 4. 핵심 변경 내용

- `MissionManagement.tsx`의 폴더 조회 결과 기본값을 렌더마다 새로 생성되는 `[]`에서 안정적인 빈 배열 상수로 변경했다.
- 서버 폴더 데이터가 실제로 달라졌을 때만 `localFolders`를 갱신하도록 `areMissionFoldersEqual()` 비교를 추가했다.
- 폴더 접기/펼치기, 폴더 순서 변경 실패 시 복구, 미션 폴더 이동에서 사용하는 기존 `localFolders` 흐름은 유지했다.
- 세부미션 목록 API가 `actionType` 관계를 함께 내려주도록 보강했다.
- `SubMissionModal.tsx`에서 액션 타입이 `신청`인 세부미션은 `검수 필요` 대신 `신청 정책 적용` 배지를 표시한다.
- 액션 타입 관계 데이터가 없거나 `신청`이 아닌 경우에는 기존처럼 `requireReview=true`일 때 `검수 필요` 배지를 표시한다.

## 5. 왜 변경했는지

기존 주제미션 탭의 폴더 동기화는 `useQuery` 기본값 배열과 `useEffect`의 무조건 `setLocalFolders()`가 결합되어 같은 데이터라도 렌더 루프가 발생할 수 있는 구조였다. 운영 화면에서 콘솔 경고가 반복되면 실제 장애와 구분이 어려워지고, 드래그/폴더 조작 중 상태 안정성도 낮아진다.

기존 DB에는 과거에 저장된 `신청 + requireReview=true` 세부미션이 남아 있다. 사용자 신청 제출 로직에서는 신청 정책이 우선 적용되지만 관리자 목록에는 `검수 필요`로 보여 운영자가 수동 검수가 적용된다고 오해할 수 있었다. DB 일괄 변경 없이 표시 정책만 실제 동작에 맞췄다.

- `git diff --check`: 통과. CRLF 변환 경고만 표시된다.
- dry-run DB 조회: `신청 + requireReview=true` 기존 데이터 23건 확인. 데이터 수정 없음.
- 브라우저 검증:
  - `PORT=5011` 임시 개발 서버에서 관리자 자동 로그인 후 `문화센터 > 미션관리 > 주제 미션` 탭 진입 확인.
  - 콘솔에서 `Maximum update depth exceeded` 경고 0건 확인.
  - 폴더 접기/펼치기 정상 동작 확인.
  - 드래그 핸들 UI 존재 확인.
  - 세부미션 관리 모달 열림 확인.
  - `신청` 세부미션은 `신청 정책 적용` 배지로 표시 확인.
  - 일반 제출 세부미션은 기존 `검수 필요` 배지 유지 확인.
  - 화면 증적: `output/playwright/theme-mission-submission-badges.png`

## 7. 주의사항 / 알려진 이슈

- 기존 `신청 + requireReview=true` 23건은 그대로 유지했다. 해당 세부미션을 다시 저장하면 기존 서버 정책에 따라 `requireReview=false`, `order=0`으로 정규화된다.
- 세부미션 목록 API 응답에 `actionType` 관계가 추가되었지만 기존 필드는 제거하지 않았다.
- 폴더 로컬 상태 비교는 `id`, `name`, `color`, `order`, `isCollapsed` 기준이다. 서버가 폴더 표시 관련 새 필드를 추가하면 비교 기준도 함께 검토해야 한다.
- 임시 개발 서버는 검증 후 종료했다.

## 9. Skill Impact Check

- 운영 정책, DB 스키마, 인증/권한, 배포 기준 변경은 없다.
- 기존 신청 정책을 관리자 목록 표시와 폴더 상태 동기화 안정화에 반영한 변경이다.
- 반복 개발 규칙이나 프로젝트 스킬 설명을 바꿀 수준의 신규 절차는 없어 스킬 업데이트는 수행하지 않았다.
