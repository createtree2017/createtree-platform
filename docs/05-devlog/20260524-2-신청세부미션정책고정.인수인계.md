# 신청 세부미션 정책 고정

작성일: 2026-05-24

## 1. 작업 개요

문화센터 관리자 세부미션 폼에서 `신청` 액션 타입의 운영 정책을 명확히 했다. `신청` 세부미션은 행사 접수/선정용으로 취급하고, 일반 제출 미션의 `표시순서` 및 `검수 필요` 설정과 혼동되지 않도록 프론트 UI와 백엔드 저장 정책을 함께 보강했다.

## 2. 연결 문서

- Plan: `docs/01-plan/features/1-20260511-세부미션_표시순서_수정.plan.md`
- Design: 없음

## 3. 변경 파일

- `client/src/components/modal/admin/SubMissionFormModal.tsx`
- `server/services/mission/mission.sub.service.ts`
- `docs/01-plan/features/1-20260511-세부미션_표시순서_수정.plan.md`

## 4. 핵심 변경 내용

- 세부미션 폼에서 액션 타입이 `신청`이면 `표시 순서` 입력을 `0`으로 고정하고 비활성화했다.
- `신청` 타입에서는 `검수 필요` 스위치를 꺼진 상태로 비활성화하고, 선착순 신청은 정원 이내 자동 승인된다는 안내를 표시한다.
- 저장 요청 전 프론트에서 `신청` 타입의 `order`와 `requireReview` 값을 각각 `0`, `false`로 정리한다.
- 백엔드 `MissionSubService`에서도 액션 타입명을 조회해 `신청` 타입 생성/수정 요청을 `order=0`, `requireReview=false`로 정규화한다.
- 기존 제출/검수 실행 로직인 `UserSubmissionService.submitSubMission()`은 변경하지 않았다.

## 5. 왜 변경했는지

주제미션이 `선착순 + 모집인원`으로 설정된 상태에서 `신청` 세부미션은 정원 이내 자동 승인, 정원 초과 대기로 처리된다. 그런데 관리자 세부미션 폼에는 `검수 필요`와 `표시 순서`가 그대로 노출되어 운영자가 수동 검수나 일반 목록 정렬이 적용된다고 오해할 수 있었다. 이번 변경은 실제 서버 정책과 관리자 설정 UX를 일치시키는 목적이다.

## 6. 동작 확인 상태

- `npm run build`: 성공
- `npm run check -- --pretty false`: 실패. 기존 `PersonaCategoryManagement.tsx`, `PersonaManagement.tsx`, `server/services/milestones.ts` 타입 오류가 남아 있으며 이번 변경 파일 관련 신규 오류는 확인되지 않았다.
- `git diff --check`: 통과. CRLF 변환 경고만 표시된다.
- API 검증:
  - 최신 코드 서버를 `PORT=5010`으로 임시 실행했다.
  - `신청` 타입 세부미션 생성 요청에서 `order=99`, `requireReview=true`를 보내도 응답은 `order=0`, `requireReview=false`로 정규화됨을 확인했다.
  - `신청` 타입 세부미션 수정 요청에서 `order=123`, `requireReview=true`를 보내도 응답은 `order=0`, `requireReview=false`로 정규화됨을 확인했다.
  - 같은 임시 세부미션을 일반 `제출` 타입으로 바꾼 뒤 `order=7`, `requireReview=true`가 그대로 저장됨을 확인했다.
- 제출 로직 검증:
  - 임시 선착순 미션(`capacity=1`, `isFirstCome=true`)에서 첫 번째 신청은 `approved`, 두 번째 신청은 `waitlist`로 처리됨을 확인했다.
  - 일반 세부미션 `requireReview=true`는 `submitted`, `requireReview=false`는 `approved`로 처리됨을 확인했다.
  - 검증용 임시 미션과 임시 세부미션은 테스트 후 삭제했다.
- 브라우저 검증:
  - `http://localhost:5010/admin?menuItem=culture-center&panel=mission-categories`에서 개발 자동 로그인 후 `테스트spspspsp` 세부미션 모달을 확인했다.
  - `신청` 세부미션 수정 화면에서 `표시 순서`가 비활성화되고 안내 문구가 표시되는 것을 확인했다.
  - `신청` 세부미션 수정 화면에서 `검수 필요` 스위치가 비활성화되고 선착순 자동승인 안내가 표시되는 것을 확인했다.
  - `제출` 세부미션 수정 화면에서는 `표시 순서`와 `검수 필요`가 기존처럼 활성 상태임을 확인했다.
  - 화면 증적: `output/playwright/submission-policy-generic.png`

## 7. 주의사항 / 알려진 이슈

- 관리자 주제미션 탭 진입 중 `MissionManagement.tsx`에서 `Maximum update depth exceeded` 콘솔 경고가 반복된다. 이번 변경 파일의 `SubMissionFormModal.tsx`가 아니라 기존 `ThemeMissionManagement` 경로에서 발생하며, 별도 안정화 작업으로 다루는 것이 좋다.
- 기존 데이터 중 `신청` 세부미션에 `requireReview=true`가 남아 있어도 사용자 신청 제출 시 선착순/모집 정책이 우선 적용된다. 이번 변경 이후 해당 세부미션을 저장하면 서버에서 정책값으로 정규화된다.
- `신청` 액션 타입 판별은 기존 시스템과 동일하게 액션 타입명 `"신청"` 기준이다. 해당 액션 타입은 수정/삭제 방어 로직이 이미 존재한다.

## 8. 연관 파일

- `server/services/mission/mission.submission.service.ts`: 사용자 신청 제출 시 선착순 자동 승인/대기 처리 로직.
- `server/services/mission/action.type.service.ts`: `"신청"` 액션 타입명 변경/삭제 방어.
- `client/src/components/modal/admin/SubMissionModal.tsx`: 세부미션 목록에서 검수 필요 배지를 표시한다.

## 9. Skill Impact Check

- 미션 운영 정책을 새로 바꾼 것이 아니라, 기존 선착순 신청 정책을 관리자 UX와 저장 정책에 반영한 변경이다.
- API 계약, DB 스키마, 배포/보안/개인정보/금전 정책 변경은 없다.
- 프로젝트 스킬의 반복 작업 규칙을 변경할 필요는 없어 스킬 업데이트는 수행하지 않았다.
