## 핵심 변경 내용

### 구현 방식: "가상 소속 전환"
- DB의 `users.hospitalId`는 **변경하지 않음** (원래 소속 보존)
- JWT에 `viewingHospitalId` 필드 주입 → 서버 미들웨어에서 이 값을 `hospitalId`처럼 사용
- 이를 통해 **로그아웃 없이 즉시 병원 전환** 가능

### 백엔드 흐름
```
superadmin이 POST /api/auth/switch-hospital { hospitalId: 5 } 호출
  → 병원 존재 여부 확인
  → 기존 JWT 디코딩 후 viewingHospitalId: 5 추가하여 새 JWT 발급
  → auth_token 쿠키 갱신 (30일)
  → 이후 모든 요청에서 requireAdminOrSuperAdmin이 effectiveHospitalId = viewingHospitalId 사용
```

### 초기화 방법
- `POST /api/auth/switch-hospital { hospitalId: null }` 호출
  - 또는 마이페이지에서 "원래 소속으로 초기화" 선택

### 프론트엔드 흐름
```
profile.tsx (마이페이지)
  → superadmin인 경우 소속 병원 영역에 [병원 전환] 드롭다운 버튼 표시
  → 클릭 시 전체 병원 목록 로드
  → 선택 시 switch-hospital API 호출 + auth_token 갱신 + /api/auth/me 캐시 무효화
```

## 버그 수정: 병원 전환 후 문화센터 데이터 미반영

### 원인
`requireAuth` 미들웨어에서 `viewingHospitalId`를 `req.user`에 별도 필드로만 전달하고, `hospitalId` 자체를 오버라이드하지 않았음. 
→ `/api/missions` 등 사용자 API가 `req.user.hospitalId`(원본 소속)로 필터링하여 전환된 병원 데이터가 조회되지 않았음.

### 수정 내용
3개 인증 미들웨어에서 superadmin + viewingHospitalId 조합 시 `hospitalId`를 오버라이드:

| 파일 | 미들웨어 | 수정 |
|------|----------|------|
| `server/middleware/auth.ts` | `requireAuth` | effectiveHospitalId 로직 추가 |
| `server/middleware/auth.ts` | `optionalAuth` | effectiveHospitalId 로직 추가 |
| `server/services/auth.ts` | `authenticateJWT` (캐시) | 캐시 히트 + DB 조회 양쪽에 effectiveHospitalId 적용 |

## 🚀 아키텍처 변경: 문화센터 페이지 전용 필터 방식으로 전환

### 배경 및 목적
기존의 JWT 병원 전환 방식(방식 A)은 "최고관리자가 모든 병원을 조회해야 하는" 본질보다 특정 병원 사용자로 위장하는 방식에 가까워 혼란을 유발할 우려가 있었습니다. 따라서, 최고관리자가 기본적으로 **모든 병원의 미션을 한눈에 볼 수 있으면서**, 필요한 경우 특정 병원만 필터링하여 볼 수 있도록 **문화센터 페이지 전용 병원 필터(방식 B)**로 리팩토링했습니다.

### 변경 내역 (3단계 완료)
1. **기존 JWT 병원 전환 코드 롤백**:
   - `middleware/auth.ts`, `services/auth.ts`의 `viewingHospitalId` 오버라이딩 로직 완벽히 제거
   - `routes/auth.ts`의 `/api/auth/switch-hospital` 엔드포인트 삭제
   - 프론트엔드의 `useHospitalSwitch.ts`, `HospitalSwitchButton.tsx` 삭제 및 `profile.tsx` 원복
2. **서버 API 필터 기능 추가 (`/api/missions`)**:
   - `user.mission.controller.ts`와 `mission.user.service.ts`에 `superadminFilter` 쿼리 파라미터 적용
   - `all`(기본값, 전체 병원+개발전용+공개조회), `hospital:{id}`(특정 병원), `dev`(개발전용), `public`(공개) 지원
3. **프론트엔드 필터 UI 추가 (`missions.tsx`)**:
   - 상단 헤더 영역에 최고관리자 전용 `<Select>` 드롭다운 제공 (`/api/hospitals` 연동)
   - 일반 회원 및 병원 관리자에게는 필터가 노출되지 않으며, 기존 방식과 동일하게 작동

### 동작 확인 상태
- ✅ 서버 재구동 및 Vite 빌드 이상 없음 (컴파일, 린트 오류 없음)
- ⬜ 수동 테스트 필수: 프로필 페이지에서 전환 기능 소실 확인, 문화센터 진입 시 "전체" 기준으로 모든 미션 표출 확인, 드롭다운 필터 변경 시 해당 미션만 리스트업되는지 확인.

## 🛡️ 추가 보완: 최고관리자 무조건 진입 권한 (프리패스)

### 발생 문제
초기 변경 후 최고관리자가 "모든 미션"을 리스트에서 볼 수는 있으나, 특정 병원에 소속된 미션의 상세 페이지에 들어가거나 서브 미션을 볼 때 **소속 병원 불일치(UNAUTHORIZED)** 또는 **선행 미션 미완료(PREVIOUS_MISSION_REQUIRED)** 에러로 튕기는 현상 발견.

### 긴급 수정 (mission.user.service.ts)
`getMissionDetail` 및 `getChildMissions` 내부 권한 검증 로직에서 `isSuperAdmin`이 `true`일 경우 모든 조건을 우회(Bypass)하도록 예외 처리.
- **HOSPITAL 타겟 미션**: 소속 병원 ID 검증 생략
- **이전/상위 미션 클리어 유무**: 제출/승인되지 않았더라도 검증 생략 (ANCESTOR, PREVIOUS, SIBLING 등 모두 포함)
- **비활성화 미션**: isActive=false 상태여도 검증 생략 (진입/조회 가능)

### 최종 확인
- 최고관리자는 어떠한 제약 조건(비활성, 병원 귀속, 상태, 선행조건)에도 걸리지 않고 모든 미션에 자유롭게 접근 및 관리 가능함을 코드로 보장 완료.
