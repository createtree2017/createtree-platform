## 자동 푸시 관리 시스템(Rule Engine) 백엔드/프론트엔드 연동 개발 완료

### 1. 개요 및 변경 배경
글로벌 CRM 표준 아키텍처인 Trigger-Audience-Payload-Toggle 구조를 도입하여, 관리자가 하드코딩 없이 유연하게 자동 푸시 규칙을 제어할 수 있는 시스템을 구축했습니다.
기존 하드코딩되었던 미션 승인/반려/상태변경 푸시 로직을 모두 Rule Engine 평가 방식으로 대체했습니다.

### 2. 변경 파일 목록
- `shared/schema.ts` (신규 테이블 `auto_push_rules` 생성 및 익스포트 연산자 추가)
- `server/services/push/push.automation.service.ts` (신규 파일: Rule 기반 푸시 발송 서비스)
- `server/controllers/push/push.automation.controller.ts` (신규 파일: 규칙 CRUD)
- `server/routes/push-automation-routes.ts` (신규 파일)
- `server/routes.ts` (라우터 등록: `/api/admin/push-automation`)
- `server/services/mission/admin.mission.review.service.ts` (기존 푸시 발송 로직 제거 및 룰 엔진 연결)
- `client/src/components/admin/PushAutomation.tsx` (신규 파일: 규칙 관리 UI)
- `client/src/pages/admin.tsx` (관리자 탭에 '자동화 설정' 메뉴 추가 연동)

### 3. 핵심 변경 내용 (왜 변경했는가?)
- **DB 스키마 추가**: 트리거(`eventType`), 대상(`hospitalId`), 템플릿(`titleTemplate`, `bodyTemplate`), 중요도 등을 저장하기 위한 테이블을 설계했습니다.
- **Rule 엔진 아키텍처 적용 (`push.automation.service.ts`)**: 
  - 이벤트 발생 시 DB에서 `isActive = true` 이면서 공통이거나(`hospitalId IS NULL`) 해당 병원에 매핑된 룰을 `우선순위(priority)` 순으로 정렬해 가장 적합한 1건을 선별합니다.
  - 관리자가 입력한 `{{userName}}`, `{{missionTitle}}` 등의 변수 칩을 동적으로 치환해 실시간 발송합니다.
  - 내부적으로 DB의 `users` 테이블을 조회해 `hospitalId` 및 `fullName`을 획득하도록 추상화하여 호출측(`admin.mission.review.service.ts`)의 부담을 낮췄습니다.
- **기존 로직 교체**: 
  - 미션 승인/반려 및 벌크 업데이트 상황에 대해 하드코딩된 `createNotification` 대신 `pushAutomationService.evaluateAndSend`를 비동기 호출토록 연결했습니다.
- **관리자 UI 컴포넌트 개발 (`PushAutomation.tsx`)**:
  - 규칙을 토글(`isActive`)하고 변수 칩버튼을 눌러 템플릿 작성의 편의를 제공하는 UI를 구축하고, `admin.tsx` 탭에 연결했습니다.

### 4. 동작 확인 상태
- `npx drizzle-kit push` 성공 (스키마 라이브 DB 전송 완료)
- ✅ 터미널 TS Type Check 검증 완료 (`npx tsc --noEmit`)
- ※ 브라우저 UI(`!!테스트!!` 명령어 미비) 검증은 보수적으로 터미널 컴파일 검증으로 통과하였습니다.

### 5. 주의사항 및 알려진 이슈
- **규칙 우선순위 문제 주의**: 병원 전용 규칙(hospitalId가 있는 항목)은 정렬 로직상 공통 규칙보다 우선 발송되도록 되어있으나, 동일 `hospitalId` 조건 내에서는 관리자가 입력한 `priority` 정수값에 따라 결정됩니다. (priority가 높을수록 우선 적용)
- **DB ReadOnly 필드 충돌 수정**: `PushAutomationController`에서 UI 폼 데이터 송신 시 포함된 `id`, `createdAt` 등의 읽기 전용 필드로 인해 Drizzle DB ORM 단계에서 `400 Bad Request` 에러 로그가 찍히던 문제를 방어했습니다. (구조 분해 할당으로 제외 처리)
- `createNetwork` / `createNotification` 을 벌크업데이트(`bulkUpdateStatus`)에서 Loop 로 다중호출하는 로직은 향후 대규모 사용자를 고려할 시 Promise.all 최적화 대상입니다.

### 6. 다음(Next Steps) 및 수동 테스트 가이드
**👤 수동 테스트 요청:**
1. `localhost:5000/api/dev/auto-login` (로그인)
2. `Admin > 앱 푸시 알림 > 자동화 설정` 탭 접속
3. **[규칙 추가] 버튼 클릭**: '미션 승인' 이벤트에 대해 기본 룰을 DB에 생성합니다. (기존 푸시 발송 조건 테스트를 위해 **최소 1개의 룰**이 켜져있어야 함!)
4. `Admin > 마일스톤 관리 > 신청내역관리` 탭에서 무작위 미션을 **승인** 또는 **반려** 처리
5. 내비 하단에서 알림이 제대로 오는 지(또는 `앱 푸시 알림 > 발송 내역` 탭에서 기록이 남는지) 확인
