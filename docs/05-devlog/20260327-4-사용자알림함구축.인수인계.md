# 20260327-4 사용자 알림함(나의 알림) 구축

## 변경 배경
FCM 푸시 시스템 구축 완료 후, 알림이 DB(`notifications` 테이블)에 저장되지만 사용자가 앱 내에서 지난 알림을 확인할 수 있는 UI가 없었음. 마이페이지 활동내역 섹션에 '나의 알림' 메뉴를 추가하고, 전용 알림 목록 페이지를 구축함.

## 변경 파일 목록

### 1. `server/routes/misc-routes.ts` — 백엔드 API 구현 (placeholder → 실제)

**기존**: `GET /api/notifications` → 빈 배열만 반환하는 placeholder  
**변경**: 4개 API 구현

| API | 메서드 | 기능 |
|-----|:------:|------|
| `/api/notifications` | GET | 사용자 본인 알림 목록 (페이징, unreadCount 포함) |
| `/api/notifications/unread-count` | GET | 읽지 않은 알림 개수만 (하단 네비 뱃지용 경량 API) |
| `/api/notifications/read/:id` | POST | 단건 읽음 처리 |
| `/api/notifications/read-all` | POST | 전체 읽음 처리 |

- 모두 `requireAuth` 미들웨어 적용
- `Promise.all`로 3개 쿼리 동시 실행 (성능)

### 2. `client/src/pages/notifications.tsx` — 알림 목록 페이지 (신규)

- `useInfiniteQuery`로 무한스크롤 구현 (기존 `snapshot-history.tsx` 패턴 준용)
- 읽지 않은 알림: 좌측 보라색 점(●) + 진한 텍스트 / 읽은 알림: 점 없음 + 흐린 텍스트
- 알림 유형별 아이콘 (승인=초록, 반려=빨강, 공지=파랑, 선물=보라)
- "모두 읽음" 버튼 (상단 우측)
- 알림 클릭 → 읽음 처리 + `actionUrl` 딥링크 이동

### 3. `client/src/pages/profile.tsx` — 마이페이지 활동내역에 메뉴 추가

- `NotificationMenuItem` 컴포넌트 신규 추가 (파일 하단)
- `/api/notifications/unread-count` 경량 API 호출 (30초 자동 갱신)
- 🔔 아이콘 + "나의 알림" 라벨 + 빨간 뱃지(99+ 처리) + "N개 안읽음" 텍스트
- 활동내역 섹션 조건문 완화 (항상 표시 — 알림 메뉴가 항상 있으므로)

### 4. `client/src/components/BottomNavigation.tsx` — MY 탭 Red Dot 뱃지

- `useQuery`로 `unread-count` API 호출 (30초 자동 갱신)
- MY 탭(`menuId === 'my-page'`) 아이콘 우측 상단에 빨간 점 뱃지 (pulse 애니메이션)
- 읽지 않은 알림 0개이면 뱃지 숨김

### 5. `client/src/hooks/usePushNotifications.ts` — FCM 포그라운드 캐시 갱신

- `useQueryClient` import 추가
- `pushNotificationReceived` 리스너 내부에 `invalidateQueries` 2줄 추가
  - `/api/notifications/unread-count` → 하단 네비 Red Dot 즉시 갱신
  - `/api/notifications` → 알림 목록 페이지 즉시 갱신

### 6. `client/src/App.tsx` — 라우트 등록

- `NotificationsPage` import 추가
- `/notifications` 라우트 등록 (ProtectedRoute + Layout)
- `getSectionTitle`에 `/notifications` → 마이페이지 섹션 매핑 추가

## 추가 수정 (초기 구현 후)

### 7. `server/routes/push-admin-routes.ts` — 관리자 푸시 → 사용자 알림함 연동

**문제**: 관리자 수동 푸시 발송 시 `pushDeliveryLogs`에만 기록되고 `notifications` 테이블에는 저장되지 않아, 사용자가 "나의 알림"에서 관리자 푸시를 볼 수 없었음.

**수정**: FCM 발송 성공 후, 대상 사용자 전원에게 `createNotification()` 비동기 호출 추가
- 전체 발송(`all`): 모든 활성 사용자ID 조회 → 알림함 저장
- 병원별 발송(`hospital`): 해당 병원 소속 사용자ID 조회 → 알림함 저장
- 개별 발송(`specific_users`): targetIds 그대로 사용 → 알림함 저장
- `Promise.allSettled`로 비동기 병렬 처리 (응답 속도에 영향 없음)
- 알림 유형: `admin_push`

### 8. `client/src/pages/notifications.tsx` — 무한 로딩 버그 수정

**문제**: API 오류 시 `isError` 상태 처리 누락 + `getNextPageParam`에서 `lastPage.pagination` 가 undefined일 때 crash
**수정**:
- `isError` 상태 추가 → 에러 시 빈 알림 화면 표시
- `retry: 1` → 3회 재시도(~7초) → 1회 재시도(~1초)로 단축
- `getNextPageParam`에 `if (!lastPage?.pagination) return undefined;` null 방어 추가

## 의사결정 이유
- **마이페이지 활동내역 하위 배치**: 새 탭이나 페이지를 추가하지 않고 기존 UI 구조를 100% 재활용
- **경량 unread-count API 분리**: 하단 네비와 마이페이지에서 30초마다 호출하므로, 목록+카운트 동시 조회보다 `SELECT count(*)` 단일 쿼리가 효율적
- **외부 라이브러리 미사용**: `react-intersection-observer` 대신 "더 보기" 버튼 방식 (기존 `snapshot-history.tsx` 패턴과 일관)
- **낙관적 업데이트 미적용 (1차)**: 단순 invalidate 패턴 우선. 실 사용 피드백 후 필요 시 2차 고도화
- **관리자/자동 푸시 통합**: 사용자 입장에서 발송 방식(수동/자동)은 무관, 모든 알림이 동일한 알림함에 표시되어야 함

## 주의사항
- `notifications` 테이블이 DB에 이미 존재해야 함 (`shared/schema.ts`에 정의됨)
- 프로덕션 배포 시 `npx drizzle-kit push` 확인 (이전 FCM 작업에서 이미 반영되어 있을 것)
- 알림 데이터가 많아지면 `notifications` 테이블에 `(user_id, is_read, created_at)` 복합 인덱스 추가 권장
- 전체 발송 시 사용자 수가 수천 명이면 `createNotification` 호출이 대량 발생 → 추후 bulk insert 최적화 필요
