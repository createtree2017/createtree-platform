# 참조 이미지 필수 보장 및 이미지 생성 로직 안정화 인수인계

작성일: 2026-04-29

## 1. 배경

아기얼굴 생성(`baby_face_img / face`)은 사용자가 업로드한 3D 초음파 이미지를 반드시 참고하여 아기 얼굴 이미지를 생성해야 하는 핵심 서비스이다.

조사 결과 `/api/generate-image` 라우트에서 Firebase 업로드로 들어온 이미지는 `req.downloadedBuffers`에 정상 다운로드되지만, 이미지 존재 판정에서 `downloadedBuffers`가 누락되어 `isTextOnlyGeneration = true`로 오판될 수 있었다.

이 문제는 아기얼굴만의 문제가 아니라 `/api/generate-image`를 사용하는 `image_upload` 컨셉 전체에서 발생할 수 있는 구조적 위험이다.

## 2. 핵심 원칙

- `generationType = image_upload`
  - 사용자 참조 이미지가 반드시 필요하다.
  - OpenAI 계열은 반드시 edit/image-to-image 경로를 사용해야 한다.
  - Gemini 계열은 반드시 `inlineData`에 참조 이미지가 포함되어야 한다.
- `generationType = text_only`
  - 이미지 없이 생성 가능하다.
  - 단, 모델 capability가 text-only를 지원해야 한다.
- 파일 존재 여부만으로 text-only 여부를 판단하지 않는다.
- Firebase URL 업로드와 Multer 파일 업로드는 서버 내부에서 동일한 `ReferenceImage[]` 구조로 정규화한다.

## 3. 완료된 작업

### Phase 1. 운영 위험 즉시 차단

- `/api/generate-image`에서 Firebase `downloadedBuffers`를 참조 이미지 판정과 AI 호출 버퍼 구성에 포함했다.
- `generationType = image_upload`인데 참조 이미지가 0개이면 AI 호출 전에 400 응답으로 차단한다.
- OpenAI image-upload 컨셉이 text-only generation 경로로 빠지지 않도록 `generationType`을 서비스까지 전달한다.
- Gemini 3.1 image-upload 컨셉이 참조 이미지가 있음에도 text-only 제한 오류를 반환하던 문제를 차단했다.

### Phase 2. 공통 이미지 입력 정규화 모듈 추가

신규 파일:

```text
server/services/image-generation/request-images.ts
server/services/image-generation/generation-mode.ts
```

`request-images.ts`는 아래 입력을 동일한 구조로 정규화한다.

- `req.files.image`
- `req.files.images`
- 기타 Multer 이미지 필드
- `req.downloadedBuffers`

정규화 구조:

```ts
type ReferenceImage = {
  buffer: Buffer;
  source: "upload" | "firebase";
  fieldName?: string;
  originalName?: string;
  mimeType?: string;
  size: number;
  index: number;
};
```

`generation-mode.ts`는 컨셉의 `generationType`, 모델 capability, 참조 이미지 개수를 기준으로 생성 모드를 결정한다.

### Phase 3. `/generate-image` 공통 모듈 적용

- 아기얼굴/만삭사진이 사용하는 `/generate-image`에 우선 적용했다.
- `referenceImages` 기준으로 단일/다중 이미지 버퍼를 구성한다.
- metadata/log에 `generationType`, `generationMode`, `referenceImageCount`, `referenceImageSources`를 기록한다.
- `image-generation-service.ts`에 image-upload 참조 이미지 누락 방어 로직을 추가했다.

## 4. 검증 결과

- `git diff --check`: 통과
  - CRLF 변환 경고만 표시됨.
- 공통 모듈 스모크 테스트: 통과
  - Firebase buffer 1개가 `source = firebase`로 정규화되는 것 확인.
  - Multer upload 1개가 `source = upload`로 정규화되는 것 확인.
  - `image_upload + referenceImageCount = 0`이면 차단 메시지 반환 확인.
- `npm.cmd run build`: 통과
  - Vite/esbuild production build 성공.
  - 기존 chunk size 및 dynamic/static import 경고만 표시됨.
- `npm.cmd run check`: 미통과
  - 이번 변경과 무관한 기존 TypeScript 오류가 다수 존재한다.
  - 신규 모듈과 `/generate-image` 변경으로 인한 빌드 오류는 확인되지 않았다.

## 5. 남은 작업

### Phase 4. `/generate-family`, `/generate-stickers` 점진 적용

- 두 라우트도 `request-images.ts`와 `generation-mode.ts`를 사용하도록 점진 적용한다.
- 다중 이미지, `imageTexts`, reference image index 매핑을 보존해야 한다.
- 현재 `/generate-stickers`는 `downloadedBuffers`를 이미지 판정에 포함하고 있으나 중복 로직이 많다.

### Phase 5. `text_only` 정책 정리

- capability registry의 `supportsTextOnly`를 기준으로 관리자 UI와 서버 검증을 일치시킨다.
- 기존 `text_only + gemini_3_1` 활성 컨셉은 운영 정책과 충돌할 수 있으므로 모델 설정을 별도로 정리한다.

### Phase 6. 어댑터 안전장치 강화

- image-upload인데 참조 이미지가 없으면 adapter 호출 전에 오류 처리한다.
- Gemini image-upload이면 `inlineData`가 반드시 포함되도록 보장한다.
- OpenAI image-upload이면 `/images/edits` 경로를 사용하도록 보장한다.
- 가능한 경우 실제 MIME 타입을 adapter에 전달한다.

### Phase 7. 최종 라우터 슬림화

최종 구조 방향:

```text
server/routes/image.ts
server/controllers/image-generation-controller.ts
server/services/image-generation/request-images.ts
server/services/image-generation/generation-mode.ts
server/services/image-generation/image-generation-service.ts
server/services/image-generation/openai-adapter.ts
server/services/image-generation/gemini-adapter.ts
```

라우터는 요청 진입과 응답만 담당하고, 이미지 입력/모드/모델 검증은 서비스 계층으로 분리한다.
