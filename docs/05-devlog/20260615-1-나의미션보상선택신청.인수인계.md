# 2026-06-15 나의미션 보상 선택 신청 인수인계

## 작업 개요

나의미션 큰미션 완료 후 사용자가 관리자가 등록한 보상 종류별 수량을 선택해 선물 신청할 수 있도록 구현했다. 보상 상품명은 코드에 고정하지 않고, 기존 관리자 보상 설정의 `giftItems`(`이미지 + 보상 이름`)를 그대로 사용자 선택 가능한 보상 종류로 사용한다.

## 핵심 변경 내용

### DB/스키마

- `userBigMissionProgress`에 `selectedRewardItem` 필드를 추가했다.
- DB 컬럼명은 `selected_reward_item`, 타입은 `jsonb`다.
- 단일 마이그레이션 SQL을 추가했다.

```sql
ALTER TABLE user_big_mission_progress
  ADD COLUMN IF NOT EXISTS selected_reward_item jsonb;
```

선택 보상은 아래 형태의 스냅샷으로 저장된다.

```json
{
  "imageUrl": "...",
  "description": "물티슈 1박스",
  "selectedIndex": 0
}
```

### 사용자 API

- `POST /api/big-missions/:id/apply-reward`가 `selectedRewardIndex`를 받도록 변경했다.
- 서버에서 다음을 검증한다.
  - 인증 사용자 여부
  - 큰미션 존재 및 활성 상태
  - 큰미션 100% 완료 여부
  - 중복 신청/이미 지급 상태 여부
  - 선택한 보상 index가 현재 미션 보상 목록에 존재하는지
- 클라이언트가 보상명이나 이미지를 임의 조작하지 못하도록 서버가 DB의 `giftItems`에서 선택 보상을 다시 찾아 스냅샷으로 저장한다.

### 사용자 화면

- `client/src/pages/mymission-detail.tsx`에서 완료 후 보상 목록을 라디오 카드로 표시한다.
- 사용자는 하나의 보상을 선택해야 `선물 신청하기` 버튼을 누를 수 있다.
- 신청 완료 또는 지급 완료 상태에서는 `selectedRewardItem`을 우선 표시한다.
- 기존 신청 건처럼 선택 스냅샷이 없으면 `선택 정보 없음`으로 표시한다.

### 관리자 화면

- `GET /api/admin/big-missions/rewards/applications` 응답에 `selectedRewardItem`을 포함했다.
- `RewardApplicationManagement` 목록에 `선택 보상` 컬럼을 추가했다.
- 선물 확인 모달에서 `사용자 선택 보상`을 최상단에 표시하고, 기존 미션 전체 보상 목록은 보조 정보로 유지했다.

### 관리자 보상 설정

- 기존 `giftItems` 구조와 입력 방식은 유지했다.
- `보상 설정` 문구를 `사용자 선택 보상`으로 보강했다.
- 자동 분류, 자동 조합, 상품별 하드코딩은 추가하지 않았다.

## 변경 이유

기존 구조는 관리자가 설정한 보상을 사용자에게 보여주고 신청만 받는 방식이었다. 사용자가 직접 보상을 선택하려면 신청 당시 선택값을 별도로 저장해야 한다. 단순 index만 저장하면 관리자가 이후 보상 목록을 수정했을 때 과거 신청 내역이 변질될 수 있으므로, 선택 당시의 `이미지 + 보상 이름`을 스냅샷으로 저장하는 방식으로 구현했다.

## 2026-06-15 추가 오류 수정 및 브라우저 검증

### 증상

- 관리자 화면에서는 `사용자 선택 보상` 설정이 정상 표시됐다.
- 사용자 화면 `/mymissions` 목록도 정상 표시됐다.
- 목록에서 큰미션 상세(`/mymissions/:id`)로 들어가면 `큰미션을 찾을 수 없습니다`가 표시됐다.
- 콘솔에는 `/api/big-missions/:id` 500 오류가 확인됐다.

### 원인

- 코드와 Drizzle schema에는 `user_big_mission_progress.selected_reward_item` 조회가 추가됐지만, 현재 실행 환경의 DB(`railway.public`)에는 해당 컬럼이 아직 없었다.
- 이 상태에서 큰미션 상세 API가 `user_big_mission_progress`를 조회하면서 500으로 실패했다.
- 별도로 상세 화면의 조회 코드가 원시 `fetch`를 사용하고 있어, 쿠키가 아닌 `localStorage.auth_token` 기반 인증에서는 목록과 상세의 인증 동작이 달라질 수 있는 취약점도 확인했다.

### 조치

- 현재 연결된 DB 대상에 단일 마이그레이션을 적용했다.

```sql

  ADD COLUMN IF NOT EXISTS selected_reward_item jsonb;
```

- 적용 전 `selected_reward_item exists before: false`, 적용 후 `selected_reward_item exists after: true`를 확인했다.
- `client/src/pages/mymission-detail.tsx` 상세 조회를 원시 `fetch`에서 공통 `apiRequest`로 변경해 Authorization 헤더와 쿠키 인증 처리를 목록/신청 API와 맞췄다.

## 주의사항 / 알려진 이슈

- 현재 연결된 DB(`railway.public`)에는 `selected_reward_item` 마이그레이션을 적용 완료했다. 다른 DB 환경에 배포할 경우 `db/migrations/20260615_add_selected_reward_item.sql` 적용 여부를 다시 확인해야 한다.

사용자가 보상 신청을 완료했더라도 관리자가 아직 지급완료 처리하지 않은 `pending` 상태라면, 배송 전까지 보상 수량이나 배송 정보를 고칠 수 있어야 한다. 반대로 관리자가 `approved`로 지급완료 처리한 뒤에는 배송/지급 업무가 확정된 상태이므로 사용자 수정은 서버와 화면 모두에서 막아야 한다.

### 변경 파일

- `server/controllers/user/user.big.mission.controller.ts`
- `client/src/pages/mymission-detail.tsx`
- `.agents/skills/createtree-aicc-platform/SKILL.md`

- `docs/02-design/features/1-20260615-나의미션_보상선택_신청.design.md`

### 핵심 변경 내용

- 보상 신청 API에서 기존 진행 row가 `pending`이면 같은 API로 수정 저장을 허용했다.
- `approved` 상태는 서버에서 `이미 지급 완료된 리워드는 수정할 수 없습니다.`로 차단한다.
- 수정 저장 시에도 기존과 동일하게 DB의 `giftItems` index 기준으로 보상과 수량을 검증한다.
- 배송 주소 필수 검증과 메모 선택값 처리도 신청과 동일하게 적용한다.
- 수정 저장 시 `rewardAppliedAt`은 최초 신청일로 유지한다.
- 수정 저장 시 `selectedRewardItem`, `rewardShippingAddress`, `rewardMemo`, `updatedAt`을 갱신한다.
- 사용자 상세 화면의 pending 보기 상태에 `신청 내용 수정` 버튼을 추가했다.
- 수정 모드에서는 기존 선택 수량, 배송 주소, 메모가 입력 UI에 채워진다.
- 수정 모드에서는 수량 합계와 배송 주소 조건이 만족될 때만 `수정 저장하기`가 활성화된다.
- 지급완료 상태에서는 수정 버튼을 표시하지 않는다.

### 주의사항 / 알려진 이슈

- pending 수정은 기존 신청 API를 재사용한다. 별도 수정 API가 아니므로 서버 상태 검증이 핵심 안전장치다.
- `rewardAppliedAt`은 최초 신청일 의미를 유지한다. 수정 시점은 현재 별도 컬럼 없이 `updatedAt`으로만 추적한다.
- 관리자 화면은 별도 구조 변경 없이 최신 DB 값을 그대로 표시한다.
- 브라우저 자동 입력 도구의 한글 입력 제약으로, 실제 브라우저 수정 검증의 주소/메모 입력은 ASCII 테스트 문자열로 수행했다. 서버/API 검증에서는 한글 주소/메모 trim 저장도 확인했다.
