# 모바일 앱 전환 방향성 보고서 (Web to App)

## 1. 개요
현재 `React + Vite`로 구축된 웹 애플리케이션을 모바일 앱(iOS/Android)으로 전환하기 위한 전략적 방향성을 분석합니다. 
최종 목표인 **React Native(완전 네이티브 UI)** 전환을 염두에 두고, **Capacitor(하이브리드 웹뷰)** 를 거쳐가는 방식의 타당성과 비용을 검증합니다.

---

## 2. 핵심 질문 분석

### Q1. Capacitor(1번)로 진행하다가 추후 React Native(2번)로 변환 가능한가?
**네, 완전히 가능합니다.** 
두 기술은 프론트엔드 아키텍처에서 'UI를 렌더링하는 방식'의 차이일 뿐입니다. 
*   **Capacitor:** 기존의 HTML/CSS 기반 React 코드를 그대로 유지한 채 껍데기만 씌웁니다.
*   **React Native:** UI 부분(`<div>`, `<span>`, CSS)을 모바일 전용 UI 컴포넌트(`<View>`, `<Text>`)로 교체합니다.
변환 시 서버 API 연동(TanStack Query), 전역 상태 관리, 데이터 정제 로직 등 '비즈니스 로직'은 React Native에서도 그대로 재사용할 수 있습니다.

### Q2. 1번을 거쳐서 2번으로 가는 것이, 지금 바로 2번을 하는 것보다 더 어려운가?
**아닙니다. 오히려 1번을 거쳐 가는 것이 비즈니스적, 기술적으로 훨씬 더 안전하고 효율적입니다.**
지금 바로 2번(React Native)으로 갈 경우의 리스크와 1번을 거칠 때의 이점은 다음과 같습니다.

| 구분 | 지금 바로 React Native (2번 직행) | Capacitor 적용 후 React Native (1번 → 2번) |
| :--- | :--- | :--- |
| **초기 앱스토어 출시** | 최소 수개월 소요 (모든 UI 재작성 필요) | **1~2주 내 완료 가능** (웹 코드 그대로 사용) |
| **초기 개발 비용** | 매우 높음 (Web / App UI 이중 관리 필요) | 매우 낮음 |
| **코드베이스 낭비** | 기존 꼼꼼하게 짠 웹 UI/CSS 코드를 일시 폐기해야 함 | 웹 모바일 환경이 개선되므로 낭비 없음 |
| **전환 난이도** | - | **동일함.** 어느 시점에 하든 UI를 React Native용으로 다시 짜는 공수는 같습니다. 1번을 했다고 해서 2번 작업이 더 어려워지지 않습니다. |

*핵심:* Capacitor 세팅에 들어가는 공수는 아주 작아서 매몰 비용(Sunk Cost)이 거의 없습니다. 반면 바로 RN으로 가면 타임투마켓(시장 출시 타이밍)을 크게 놓치게 됩니다.

### Q3. 최종 목적지가 React Native라면 어떻게 접근해야 하는가?
최종 목적지가 RN이라면 더욱 **"1번(Capacitor) 적용 → 앱스토어 출시 및 시장 검증 → 2번(React Native) 점진적 마이그레이션"** 전략을 강력히 권장합니다.

## 3. 전략적 방향성 및 실행 로드맵 (추천)

### 1단계: Capacitor로 빠른 시장 진입 (소요 시간: 1~2주)
*   **목표:** 최소한의 수정으로 iOS/Android 마켓에 앱을 등록하고 고객 피드백 수집 시작.
*   **작업 내용:** 
    1. Capacitor 라이브러리 설치 및 세팅
    2. 푸시 알림, 카메라 권한 등 네이티브 필수 기능만 플러그인 연동
    3. 모바일 브라우저 최적화 (Safari/Chrome 100% 대응) 및 터치 영역 개선
*   **이점:** 앱스토어 심사 프로세스 사전 경험, 빠른 고객 유입.

### 2단계: 비즈니스 로직 분리 (RN 전환 준비기)
*   **목표:** 추후 RN 전환 시 UI만 걷어내기 쉽도록 코드 구조 개선.
*   **작업 내용:**
    *   화면(UI) 파일과 로직(API, 계산) 파일을 철저히 분리. (Custom Hook 적극 활용)
    *   웹에서 기능 업데이트를 지속하면서, 로직 재사용성을 극대화합니다.

### 3단계: React Native 전환 (본격적인 네이티브 앱 개발)
*   **목표:** 앱 사용자가 많아지고 고성능/부드러운 애니메이션이 필수불가결해지는 시점.
*   **작업 내용:**
    *   **Expo / React Native Web** 등 모노레포 아키텍처 도입 (웹과 앱 코드 통합 관리 목적).
    *   준비해둔 비즈니스 로직(Hook)을 가져와서 RN용 UI(`<View>`)에 이어 붙이는 작업 진행.

---

## 4. 최종 결론

> **1번(Capacitor)으로 먼저 진행했다가 2번(React Native)으로 가는 것은 전혀 문제가 없을 뿐만 아니라, 훌륭한 IT 스타트업들이 채택하는 가장 표준적이고 린(Lean)한 개발 방법론입니다.**

당장 디자인과 로직이 수시로 바뀔 수 있는 초기 단계에서 바로 RN을 시작하면 개발 피로도가 극심해집니다. 웹 코드를 하나로 관리하면서 빠르게 시장에 런칭(Capacitor)한 후, 기능이 충분히 고도화되고 안정화되었을 때 네이티브(React Native)로 마이그레이션하는 방향을 추천합니다.
