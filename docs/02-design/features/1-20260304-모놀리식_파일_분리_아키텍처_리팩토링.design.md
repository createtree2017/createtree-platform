# 🏗 모놀리식 파일 아키텍처 분리 설계서 (PDCA - Design)

> **작성일**: 2026-03-04
> **대상 지점**: `server/routes/mission-routes.ts`, `MissionManagement.tsx`, `ImageGenerationTemplate.tsx`

이 문서는 모놀리식 구조(거대 파일)를 "의존성 분리원칙(SRP)"에 따라 분할하는 구체적인 경로와 파일 구조를 설계합니다.

## 🚀 1. 백엔드 분리 아키텍처 (Backend Layering)

가장 거대한 `server/routes/mission-routes.ts` (5132줄)를 다음의 디렉토리 구조로 분리하여 레이어드 아키텍처를 구현합니다.

### 📂 디렉토리 구조 변경안

```text
server/
├── controllers/          # (NEW) HTTP 파라미터 파싱 및 검증, 에러 응답
│   ├── admin/
│   │   ├── admin.mission.controller.ts
│   │   └── admin.mission.category.controller.ts
│   └── public/
│       ├── mission.participation.controller.ts
│       └── mission.review.controller.ts
│
├── services/             # (NEW) 핵심 비즈니스 로직, Drizzle DB 트랜잭션, 타 모듈 연동
│   ├── mission/
│   │   ├── mission.category.service.ts
│   │   ├── mission.core.service.ts
│   │   ├── mission.participation.service.ts
│   │   └── mission.tree.service.ts  # (빌더, 구조 병합)
│
└── routes/
    ├── mission-routes.ts # (MODIFIED) 로직 없이 라우트 매핑 + 권한 미들웨어만 존재
```

### 🧩 1-1. Routes 변경 예시 (Before vs After)

**[Before: `mission-routes.ts`]** (기존 덩어리)

```typescript
router.post('/admin/mission-categories', requireAdmin, async (req, res) => {
  try {
    const data = schema.parse(req.body);
    // [비즈니스 로직 50줄...]
    const [result] = await db.insert(...);
    // [외부 연동 로직...]
    res.json(result);
  } catch(e) { ... res.status(500) ... }
});
```

**[After: `mission-routes.ts`]** (의존성 분리)

```typescript
import { AdminMissionCategoryController } from "../controllers/admin/admin.mission.category.controller";

const categoryController = new AdminMissionCategoryController();

// 라우터는 경로와 파라미터 매핑만 주관
router.post(
  "/admin/mission-categories",
  requireAdmin,
  categoryController.createCategory,
);
```

## 💻 2. 프론트엔드 분리 아키텍처 (Frontend View-Logic Split)

프론트엔드의 `MissionManagement.tsx` (3611줄)와 `ImageGenerationTemplate.tsx` (1440줄)는 다음 원칙에 따라 "커스텀 훅"과 "서브 컴포넌트"로 추출합니다.

### 📂 디렉토리 구조 변경안

```text
client/src/
├── hooks/
│   ├── queries/                  # (NEW) TanStack Query 관련 훅
│   │   ├── useMissionQueries.ts
│   │   └── useImageGenerationQueries.ts
│   └── features/                 # (NEW) 복합 상태 관리 로직
│       ├── useMissionManagement.ts
│       └── useImageUploadStore.ts
│
└── components/
    ├── admin/
    │   └── mission/              # (NEW) 미션 관리자 하위 UI 쪼개기
    │       ├── MissionFilterBar.tsx
    │       ├── MissionTable.tsx
    │       └── MissionForms/     # 생성, 수정 포맷 등
    │
    └── image-generation/         # (NEW) 제너레이터 템플릿 하위 UI
        ├── StyleSelectorDialog.tsx
        ├── MultiImageUploader.tsx
        └── GenerationResultView.tsx
```

### 🧩 2-1. `MissionManagement.tsx` 쪼개기 명세

- **상태/로직 분리**: `useMissionManagement.ts`를 생성하여 정렬 상태(sort), 페이지네이션 상태, 선택된 행(selected rows), 삭제 / 승인 핸들러를 위임합니다.
- **UI 분리**: `MissionManagement.tsx`는 이 훅을 호출하여 상태를 주입받은 뒤, `<MissionFilterBar />` 와 `<MissionTable />` 조합으로만 화면을 렌더링하도록 뼈대를 가볍게 만듭니다. (파일 길이를 500줄 이하로 타겟팅)

### 🧩 2-2. `ImageGenerationTemplate.tsx` 쪼개기 명세

- 가장 무거운 부분인 **"Firebase / GCS 파일다중 업로드 상태 업데이트 및 진행률 바"** 관리 로직을 순수 TypeScript 훅(`useImageUploadStore.ts` 등)으로 완전 매핑합니다.
- UI 컴포넌트에는 하드코딩된 Tailwind Color(`bg-gray-800` 등)를 전부 배제하고 테마 변수(`bg-card`, `text-card-foreground`)를 주입하여 라이트/다크 모드 불일치를 해소합니다.
