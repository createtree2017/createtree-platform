# FCM 디바이스 토큰 백엔드 연동 설계서 (Design)

## 1. 개요
*   **목적:** 사용자의 기기(Web, Android, iOS)에서 생성된 FCM 푸시 토큰을 수집하여 푸시 알림을 정상적으로 발송하기 위한 인프라 구축.
*   **플랫폼 호환성(Cross-Platform):** 향후 배포 단계(1단계: 자체 APK 배포 -> 2단계: Android 스토어 분리 -> 3단계: iOS 스토어 확장)를 모두 고려하여, 특정 환경에 종속되지 않는 범용적인 구조로 설계합니다. 현재 웹 사용자(Web Push) 환경에도 전혀 영향을 주지 않고 호환되도록 만듭니다.

## 2. 데이터베이스 설계 (`shared/schema.ts`)

단일 사용자가 여러 기기(예: 스마트폰, 태블릿, PC 웹브라우저)에서 동시 접속하거나 푸시 알림을 받을 수 있는 멀티-디바이스 환경을 기본적으로 지원합니다.

*   **테이블명:** `user_devices`
*   **스키마 구조:**
    ```typescript
    export const userDevices = pgTable("user_devices", {
      id: serial("id").primaryKey(),
      userId: integer("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
      deviceToken: text("device_token").notNull().unique(), // FCM에서 발급된 고유 토큰 값
      deviceType: varchar("device_type", { length: 20 }).notNull().default("unknown"), // 'android', 'ios', 'web'
      isActive: boolean("is_active").notNull().default(true), // 사용자가 권한을 끄거나 앱을 지웠을 때 false 처리용
      lastUsedAt: timestamp("last_used_at").defaultNow().notNull(), // 토큰이 가장 마지막으로 서버에 보고(갱신)된 시간
      createdAt: timestamp("created_at").defaultNow().notNull(),
    });

    // Zod 스키마 검증용
    export const insertUserDeviceSchema = createInsertSchema(userDevices);
    export type UserDevice = typeof userDevices.$inferSelect;
    export type InsertUserDevice = z.infer<typeof insertUserDeviceSchema>;
    ```
*   **관계(Relations):**
    *   `users` 1 : N `user_devices` 구조 (하나의 유저가 여러 디바이스 토큰 보유 가능)

## 3. 백엔드 API 설계 (`server/routes.ts`)

프론트엔드(웹 브라우저 및 하이브리드 앱 내부)에서 발급받은 FCM 토큰을 안전하게 DB에 동기화합니다.

*   **엔드포인트:** `POST /api/users/device-token`
*   **보안:** 인증된 사용자(User 로그인 세션 필수)만 호출 가능 (`isAuthenticated` 미들웨어 사용)
*   **Request Body Type:**
    ```typescript
    {
      token: string;       // (필수) FCM 고유 토큰
      deviceType: string;  // (선택) 'android' | 'ios' | 'web' (기본값: 'web')
    }
    ```
*   **비즈니스 처리 로직 흐름 (Upsert 로직):**
    1.  클라이언트로부터 토큰 전달받음
    2.  DB의 `user_devices`에서 `deviceToken` 기준으로 검색
    3.  **경우 1 (토큰이 없을 때):** 새로운 기기로 판단하여 신규 `Insert` (현재 접속 중인 `userId`로 맵핑)
    4.  **경우 2 (토큰이 있지만, `userId`가 다를 때):** 폰을 다른 사람에게 양도했거나, 기기에서 다른 계정으로 로그인한 경우. 기존 토큰 소유자를 현재 로그인한 `userId`로 **변경(Update)**
    5.  **경우 3 (토큰이 있고, `userId`도 내 것일 때):** 단순 앱 재실행. `lastUsedAt` 값만 최신 시간으로 **업데이트(Update)**하여 현재 토큰이 "활성 상태" 임을 증명함.
    6.  성공 여부 Response (예: `200 OK`)

## 4. 프론트엔드 연동 설계

기존 작업물인 `client/src/lib/push-notifications.ts` (푸시 알림 훅/유틸)에서 토큰을 획득하는 부분의 직후에 API 호출 로직을 주입합니다.

*   **호출 시점:** 앱 로딩/로그인 직후 Capacitor FCM 플러그인에서 `token` 이벤트 콜백이 발생할 때 전달.
*   **플랫폼 감지 로직:** Capacitor의 `Capacitor.getPlatform()` API를 사용하여, 현재 이 접속이 `'web'`, `'android'`, `'ios'` 중 어디인지 자동으로 식별하여 `deviceType`을 API Payload에 실어 보냅니다. (이로 인해 앱과 웹을 독립적으로 타겟팅 가능)

## 5. 결론 및 진행 방향
이 설계는 현재 Web 기반 사용자들에게는 어떤 데이터 변화나 장애도 유발하지 않는 **안전한 구조(비파괴적인 확장 구조)**입니다.
또한 1단계(자체 APK) 이후 2, 3단계(안드로이드 스토어, iOS 스토어) 배포 시에도 스키마를 전혀 수정할 필요 없이 그대로 대응 가능한 **모바일 표준 토큰 수집 패턴**입니다.

이 설계서를 바탕으로 `/check` (설계 대비 구현)를 위한 로직 작성으로 바로 들어가겠습니다.
