# 1. 설계 목표

나의미션 완료 기준을 운영 방식에 맞게 고정한다. 큰미션 topic은 특정 `themeMissionId`를 들고 있지 않고 `categoryId`만 가진다. 사용자가 같은 카테고리의 문화센터 프로그램 중 하나를 끝까지 완료하면 topic이 완료된다.

핵심은 “끝까지 완료”의 정의를 `신청 승인 1건`이 아니라 `해당 프로그램의 활성 세부미션 전체 승인`으로 바꾸는 것이다.

# 2. 완료 계산 서비스

새 서비스 파일:

```text
server/services/mission/big-mission-progress.service.ts
```

서비스 책임:

- 사용자별 완료 category map 계산
- 큰미션 topic 평가
- 큰미션 목록/상세 payload에 필요한 progress 계산
- `user_big_mission_progress` 저장값 재계산

완료 category SQL 기준:

```sql
theme_missions.category_id = big_mission_topics.category_id
AND count(active sub_missions) > 0
AND count(active sub_missions) = count(distinct approved submissions by user for active sub_missions)
```

`theme_missions.is_active`는 완료 후보에서 강제하지 않는다. 운영상 월별/회차별 프로그램이 비활성화되어도 사용자의 과거 완료 이력은 카테고리 완료로 인정될 수 있기 때문이다. 단, 세부미션은 “관리자가 현재 활성화한 세부미션”만 분모로 사용한다.

# 3. API 적용

## 3.1 `GET /api/big-missions`

- 기존 `approvedSubmissions -> approvedCategoryMap` 로직 제거
- 공통 서비스의 `calculateMissionsProgress` 사용
- 응답의 `completedTopics`, `totalTopics`, `progressPercent`, `status`는 활성 topic 기준으로 계산

## 3.2 `GET /api/big-missions/:id`

- 공통 서비스의 `calculateMissionProgress` 사용
- topic별 `isCompleted`, `completedMissionTitle`, `completedThemeMissionId` 포함
- 기존 reward fields는 유지

## 3.3 `POST /api/big-missions/:id/apply-reward`

- 보상 선택/배송 정보 검증은 기존 정책 유지
- 신청 전 공통 서비스 기준으로 100% 완료 여부를 검증
- `cancelled` 이력이 있어도 새 기준으로 완료 상태면 재신청 가능
- `approved`는 수정/재신청 차단
- pending은 기존처럼 수정 가능

# 4. 저장 progress 재계산

관리자 세부미션 승인/반려 후 `admin.mission.review.service.ts`에서 공통 서비스의 재계산 메서드를 호출한다.

재계산 저장 정책:

- `completedTopics`, `totalTopics`, `status`, `completedAt`, `updatedAt` 갱신
- reward 관련 필드는 자동 삭제하지 않는다.
- 기존에 pending/approved 보상 신청이 있는데 새 기준으로 미완료가 되더라도 신청 상태는 유지한다.
- 운영자는 dry-run 결과를 보고 pending은 운영취소, approved는 수동 검토한다.

# 5. 문화센터 진행률 보정

문화센터 프로그램 진행률은 화면별로 계산 지점이 여러 곳에 있다. 모든 지점에서 다음 기준을 적용한다.

- denominator: `sub_missions.is_active=true`
- numerator: `sub_mission_submissions.status='approved'`이면서 해당 submission의 subMission도 `is_active=true`

이 기준을 만족시키기 위해 approved count 쿼리의 `IN (SELECT id FROM sub_missions WHERE theme_mission_id=...)` 조건에 `is_active=true`를 추가한다.

# 6. 보상 신청 운영취소

## 6.1 DB

`user_big_mission_progress`에 취소 이력을 추가한다.

```ts
rewardCancelledAt: timestamp
rewardCancelledBy: integer references users.id
rewardCancelReason: text
```

`rewardStatus`는 기존 varchar라 enum migration 없이 `cancelled` 값을 사용한다.

## 6.2 API

```text
POST /api/admin/big-missions/rewards/:progressId/cancel
```

요청:

```json
{
  "reason": "새 완료 기준상 문화센터 프로그램 전체 완료 전 신청으로 확인"
}
```

정책:

- `pending`만 운영취소 가능
- `approved`는 400 응답으로 수동 검토 안내
- 취소 시 `rewardStatus='cancelled'`, 취소일/취소자/사유 저장
- 사용자가 이후 새 기준으로 완료하면 다시 신청 가능

# 7. 관리자 UI

`RewardApplicationManagement.tsx` 변경:

- 상태 필터에 `cancelled` 추가
- 상태 badge 추가
- pending 행에 운영취소 버튼 추가
- cancelled 행은 취소일/사유 표시
- approved 행은 자동취소하지 않는 수동 검토 상태로 유지

# 8. dry-run 스크립트

스크립트:

```text
scripts/big-mission-completion-dry-run.ts
```

출력 항목:

- 사용자 ID/이름/전화/이메일/병원명
- 큰미션 ID/미션명
- 기존 완료 상태와 완료 topic 수
- 새 기준 완료 상태와 완료 topic 수
- rewardStatus, rewardAppliedAt
- 완료에서 미완료로 내려갈 대상
- pending/approved 보상 신청 취소/수동검토 대상

스크립트는 조회만 수행하고 DB를 변경하지 않는다.

# 9. 리스크와 대응

- 기존 보상 신청 데이터가 새 기준으로 미완료가 될 수 있다.
  - 자동 취소하지 않고 dry-run과 관리자 운영취소로 분리한다.
- approved 지급완료 건은 이미 외부 배송/지급 업무와 연결되었을 수 있다.
  - 자동 취소하지 않고 수동 검토 대상으로 출력한다.
- 비활성 세부미션의 과거 approved 제출이 남아 있을 수 있다.
  - 모든 진행률 numerator에서 활성 세부미션 조건을 강제한다.
- 기존 타입 오류가 남아 있다.
  - `npm run check -- --pretty false` 결과에서 기존 오류와 이번 변경 파일 오류를 구분해 보고한다.
