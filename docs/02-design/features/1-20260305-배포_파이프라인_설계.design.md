# 디자인 설계서: 간소화된 2단계 배포 파이프라인 (Dev -> Prod)

- **상태**: 📝 Draft (Revised)
- **작성일**: 2026-03-05
- **관련 기능**: CI/CD 간소화, 배포 안정성 유지

---

## 1. 개요

복잡한 절차를 줄이고 **'개발(Dev)'**과 **'정식(Prod)'** 두 단계로만 구성된 직관적인 배포 파이프라인을 설계합니다. 개발 환경에서 자유롭게 테스트하고, 검증된 코드만 정식 서비스에 반영하는 구조입니다.

## 2. 배포 아키텍처 (2단계)

### 2-1. 환경 구성 (Railway Environments)

| 구분 | 환경명 | 대상 브랜치 | 주요 용도 |
|------|--------|------------|----------|
| **1단계: 개발 (Dev)** | `develop` | `develop` | 기능 개발, 통합 테스트, 중간 점검 |
| **2단계: 정식 (Prod)** | `production` | `main` | 실제 사용자 서비스 운영 (`createtree.ai.kr`) |

### 2-2. 브랜치 운영 방식

1. **`develop`**: 모든 개발 작업이 모이는 곳입니다. 푸시 즉시 개발 서버에 반영됩니다.
2. **`main`**: 정식 서비스용 브랜치입니다. `develop`의 내용이 완벽할 때만 머지합니다.

---

## 3. 단순화된 워크플로우

### Step 1: 개발 및 자동 배포
- `develop` 브랜치에 코드를 푸시하거나 PR을 머지합니다.
- **Railway Dev 환경**이 즉시 자동으로 빌드 및 배포됩니다.
- 개발 서버에서 기능이 정상인지 확인합니다.

### Step 2: 정식 배포 승인 (Prod)
- 개발 서버 검증이 끝나면 `develop` 브랜치를 `main` 브랜치로 머지합니다. (PR 생성 권장)
- **`main` 머지가 곧 '정식 배포 승인'을 의미합니다.**
- 머지 즉시 **Railway Prod 환경**이 업데이트되어 정식 서비스에 적용됩니다.

---

## 4. 기대 효과

- **단순함**: 복잡한 Staging 단계 없이 Dev-Prod 2단계로 빠른 배포 가능.
- **안정성**: `main` 브랜치를 직접 수정하지 않으므로 정식 서비스 장애 예방.
- **자동화**: 브랜치 머지만으로 배포가 완료되는 편리한 환경 제공.
