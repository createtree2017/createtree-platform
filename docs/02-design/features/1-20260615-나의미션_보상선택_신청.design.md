# 1. 설계 목표

큰미션 완료 사용자가 관리자가 등록한 보상 종류별 수량을 선택해 선물 신청하도록 한다. 보상 상품은 운영 중 언제든 바뀔 수 있으므로, 코드는 상품 종류를 알거나 자동 조합하지 않는다. 관리자가 입력한 `이미지 + 보상 이름`만 선택 가능한 보상 종류로 사용한다.

2026-06-15 추가 확장에서는 `하나 선택` 방식에서 `보상 종류별 수량 선택` 방식으로 변경한다. 관리자는 보상 조합을 등록하지 않고 보상 종류와 총 선택 가능 수량만 설정하며, 사용자는 모바일에서 `+ / -`로 수량을 골라 합계를 맞춘다. 보상 신청 시 배송 받을 주소를 필수로 받고, 사용자 메모는 선택값으로 함께 저장한다. 관리자가 아직 지급완료 처리하지 않은 `pending` 상태에서는 사용자가 신청한 수량, 주소, 메모를 수정할 수 있게 한다.

# 2. 데이터 모델

## 2.1 보상 선택지 원천

기존 큰미션의 `giftItems`를 그대로 사용한다.

```ts
giftItems: Array<{
  imageUrl: string;
  description: string;
}>
```

레거시 데이터 호환을 위해 `giftItems`가 비어 있고 `giftImageUrl` 또는 `giftDescription`이 있으면 단일 선택지로 취급한다.

## 2.2 신청 스냅샷

`user_big_mission_progress.selected_reward_item`에 신청 당시 보상을 저장한다.

```ts
type SelectedRewardItem = {
  imageUrl?: string;
  description: string;
  selectedIndex: number;
};
```

`selectedIndex`는 관리자가 등록한 `giftItems` 배열의 index다. 스냅샷은 신청 후 관리자 보상 설정이 바뀌어도 신청 내역을 보존하기 위한 값이므로, 관리자 화면은 이 값을 우선 표시한다.

수량 선택 신청은 기존 `selected_reward_item` 컬럼을 배열형 스냅샷으로 확장해 저장한다. 기존 단일 객체 데이터와 새 배열 데이터를 모두 읽는다.

```ts
type SelectedRewardQuantityItem = {
  imageUrl?: string;
  description: string;
  selectedIndex: number;
  quantity: number;
};

type SelectedRewardSnapshot =
  | SelectedRewardItem
  | SelectedRewardQuantityItem[];
```

큰미션별 총 선택 가능 수량은 `big_missions.reward_selection_limit`에 저장한다. 기존 미션의 기본값은 1로 둬 기존 단일 선택 동작을 유지한다.

## 2.3 배송 정보

보상 신청 시 배송 정보를 `user_big_mission_progress`에 함께 저장한다.

```ts
rewardShippingAddress?: string | null;
rewardMemo?: string | null;
```

- `reward_shipping_address`: 배송 받을 주소, 신청 시 필수
- `reward_memo`: 사용자 메모, 선택값
- 기존 신청 건은 두 값이 `null`일 수 있으므로 화면에서 fallback 문구를 제공한다.

# 3. API 계약

## 3.1 사용자 큰미션 상세

`GET /api/big-missions/:id`

응답에 추가:

```ts
selectedRewardItem?: SelectedRewardSnapshot | null;
rewardSelectionLimit: number;
rewardShippingAddress?: string | null;
rewardMemo?: string | null;
```

## 3.2 보상 신청

`POST /api/big-missions/:id/apply-reward`

요청:

```json
{
  "selectedRewardIndex": 0
}
```

수량 선택 요청:

```json
{
  "selectedRewards": [
    { "selectedIndex": 0, "quantity": 1 },
    { "selectedIndex": 1, "quantity": 2 }
  ],
  "shippingAddress": "서울시 ...",
  "rewardMemo": "부재 시 문 앞에 놓아주세요"
}
```

서버 검증:

- 인증된 사용자만 가능
- 큰미션이 활성 상태여야 함
- 승인된 주제미션 카테고리 기준으로 진행률 100%여야 함
- 기존 `rewardStatus`가 `pending` 또는 `approved`면 거절
- `selectedRewardIndex`는 정수여야 함
- 해당 index의 보상 선택지가 존재하고 이미지 또는 보상명이 있어야 함
- `selectedRewards`의 `selectedIndex`는 정수여야 함
- `quantity`는 0 이상 정수여야 함
- 선택 수량 합계는 `rewardSelectionLimit`와 정확히 같아야 함
- 전부 0개 선택은 거절
- `shippingAddress`는 trim 후 빈 값이면 거절
- `rewardMemo`는 선택값이며, trim 후 빈 값이면 `null` 저장
- 기존 진행 row가 `pending`이면 같은 API로 수정 저장 허용
- 기존 진행 row가 `approved`이면 수정 또는 재신청 거절

저장:

```ts
{
  rewardStatus: "pending",
  rewardAppliedAt: new Date(),
  selectedRewardItem: {
    imageUrl,
    description,
    selectedIndex
  }
}
```

수량 선택 저장:

```ts
{
  rewardStatus: "pending",
  rewardAppliedAt: new Date(),
  selectedRewardItem: [
    { imageUrl, description, selectedIndex, quantity }
  ],
  rewardShippingAddress,
  rewardMemo
}
```

pending 수정 저장:

```ts
{
  rewardStatus: "pending",
  rewardAppliedAt: existingProgress.rewardAppliedAt,
  selectedRewardItem: [
    { imageUrl, description, selectedIndex, quantity }
  ],
  rewardShippingAddress,
  rewardMemo,
  updatedAt: new Date()
}
```

`rewardAppliedAt`은 최초 신청일 의미를 유지하기 위해 수정 시 덮어쓰지 않는다.

## 3.3 관리자 신청 목록

`GET /api/admin/big-missions/rewards/applications`

응답에 추가:

```ts
selectedRewardItem?: SelectedRewardSnapshot | null;
rewardShippingAddress?: string | null;
rewardMemo?: string | null;
```

# 4. UI 설계

## 4.1 사용자 화면

파일: `client/src/pages/mymission-detail.tsx`

- `giftItems` 또는 레거시 보상으로 `rewardChoices`를 만든다.
- 완료 전에는 기존 보상 목록을 잠금 상태로 보여준다.
- 완료 후 신청 전에는 각 보상을 수량 선택 가능한 카드로 표시한다.
- 2026-06-15 수량 선택 확장 후에는 각 보상을 모바일 터치용 `+ / -` 스테퍼로 표시한다.
- 현재 선택 합계와 총 선택 가능 수량을 상단에 표시한다.
- 합계가 총 선택 가능 수량과 정확히 같아야 신청 버튼을 누를 수 있다.
- 수량 선택 아래에 배송 받을 주소와 메모 입력란을 배치한다.
- 배송 주소가 비어 있으면 신청 버튼을 비활성화한다.
- 메모는 선택 입력으로 두며 빈 값 저장은 `null`로 통일한다.
- 신청 완료 또는 지급 완료 상태에서는 `selectedRewardItem`을 우선 표시한다.
- 신청 완료 또는 지급 완료 상태에서는 배송 주소와 메모도 함께 표시한다.
- 신청 완료 상태가 `pending`이면 보기 모드에서 `신청 내용 수정` 버튼을 표시한다.
- 수정 모드에서는 기존 `selectedRewardItem`, `rewardShippingAddress`, `rewardMemo`를 입력 UI에 다시 채운다.
- 수정 저장 버튼은 수량 합계가 `rewardSelectionLimit`와 같고 배송 주소가 있을 때만 활성화한다.
- `approved` 지급 완료 상태에서는 수정 버튼을 표시하지 않는다.
- 기존 단일 스냅샷은 수량 1개로 표시한다.
- 새 배열 스냅샷은 보상별 수량을 함께 표시한다.
- 기존 신청처럼 선택 스냅샷이 없으면 기존 신청 안내와 현재 등록된 보상 목록을 표시한다.

## 4.2 관리자 신청 관리

파일: `client/src/components/admin/RewardApplicationManagement.tsx`

- 목록에 선택 보상 요약을 추가한다.
- 선물 확인 모달은 "사용자 선택 보상"을 최상단에 표시한다.
- 새 배열 스냅샷은 `물티슈 1개 + 기저귀 2개`처럼 표시한다.
- 기존 단일 스냅샷은 수량 1개로 표시한다.
- `selectedRewardItem`이 없으면 선택형 수량 적용 전 기존 신청 건으로 표시한다.
- 배송 주소가 없으면 `배송 정보 없음`, 메모가 없으면 `메모 없음`으로 표시한다.
- 선물 확인 모달은 사용자 선택 보상, 배송 정보, 미션 전체 보상을 구분해 표시한다.
- 필요 시 아래에 "미션에 등록된 전체 보상"을 보조 정보로 표시한다.

## 4.3 관리자 보상 설정

파일: `client/src/components/modal/admin/BigMissionModals.tsx`

- 기존 `giftItems` 입력 구조를 유지한다.
- 문구를 "사용자 선택 보상"으로 보강해, 등록한 항목이 사용자가 수량을 고를 수 있는 보상 종류라는 점을 명확히 한다.
- `rewardSelectionLimit` 입력을 추가해 총 선택 가능 수량을 설정한다.
- 자동 조합이나 상품별 하드코딩 필드는 추가하지 않는다.

# 5. 마이그레이션 설계

단일 SQL 마이그레이션을 준비한다.

```sql
ALTER TABLE user_big_mission_progress
ADD COLUMN IF NOT EXISTS selected_reward_item jsonb;

ALTER TABLE big_missions
ADD COLUMN IF NOT EXISTS reward_selection_limit integer NOT NULL DEFAULT 1;

ALTER TABLE user_big_mission_progress
ADD COLUMN IF NOT EXISTS reward_shipping_address text;

ALTER TABLE user_big_mission_progress
ADD COLUMN IF NOT EXISTS reward_memo text;
```

이번 구현에서는 마이그레이션 파일을 준비하고 현재 연결된 개발 DB에 아래 컬럼 적용을 확인했다.

- `user_big_mission_progress.selected_reward_item`
- `big_missions.reward_selection_limit`
- `user_big_mission_progress.reward_shipping_address`
- `user_big_mission_progress.reward_memo`

다른 배포 환경에는 배포 전 동일 마이그레이션 적용 여부를 다시 확인해야 한다.

# 6. 리스크와 대응

- 기존 `pending`/`approved` 신청에는 선택 스냅샷이 없다.
  - 관리자와 사용자 화면에서 기존 신청 안내 fallback을 둔다.
- 기존 단일 선택 스냅샷과 새 배열 스냅샷이 공존한다.
  - 화면 표시 helper는 단일 객체를 수량 1개로 정규화한다.
- 관리자가 보상을 수정한 뒤 사용자가 오래된 화면에서 신청할 수 있다.
  - 서버에서 현재 DB의 `giftItems` index를 다시 검증한다.
- 클라이언트가 수량을 조작할 수 있다.
  - 서버에서 수량 정수성, index 존재 여부, 합계 일치를 다시 검증한다.
- 빈 보상 항목이 등록될 수 있다.
  - 이미지와 설명이 모두 비어 있는 항목은 선택지에서 제외한다.
- DB 컬럼 미적용 상태로 배포하면 신청/조회 API가 실패할 수 있다.
  - 배포 전 단일 마이그레이션 적용 여부를 확인한다.
- 배송 주소는 개인정보 성격이 있으므로 관리자 신청 관리 화면에서 필요한 위치에만 표시하고, 기존 신청 건은 빈 값으로 안전하게 처리한다.
