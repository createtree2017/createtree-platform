> 이 문서는 대형 파일(1000줄+) 분리 시 **반드시 읽고 따라야 하는 필수 절차**입니다.

## 1. 우리 사이트의 특수성 (createTree 고유 위험 요소)

### 1.1 Express + Vite SPA Fallback 구조

```
요청 흐름:
클라이언트 → Express API 라우트 → (매칭 없으면) → Vite SPA Fallback → HTML 반환
```

**위험**: API 라우트가 누락되면 404가 아닌 **HTML 페이지가 반환됨**.
프론트엔드는 JSON을 기대하므로 **파싱 에러 발생 → 화면 빈칸 표시**.
이 구조 때문에 라우트 누락이 즉시 발견되지 않고, 사용자가 문제를 보고할 때까지 모를 수 있음.

### 1.2 라우트 등록 순서 민감성

```typescript
// server/routes.ts에서의 등록 순서가 중요
registerAdminRoutes(app);    // 1. 관리자 라우트
app.use('/api', missionRoutes);  // 2. 미션 라우트
// ... (이하 모든 API 라우트)
// Vite SPA fallback은 반드시 마지막
```

**위험**: 라우트를 분리하면서 등록 순서가 바뀌거나 누락되면, 특정 API가 SPA fallback에 잡혀서 HTML을 반환함.

### 1.3 서버 Hot Reload 미반영

개발 서버(`npm run dev`)에서 **서버 사이드 코드 변경은 자동 반영되지 않는 경우가 있음**.
특히 새 파일 추가, import 경로 변경, 라우트 등록 변경은 서버 재시작 필수.

### 1.4 현재 대형 파일 현황 (2026-03-05 기준)

| 파일 | 줄 수 | 위험도 | 분리 우선순위 |
|------|-------|--------|-------------|
| `client/src/components/admin/MissionManagement.tsx` | 3,591줄 | 🔴 매우 높음 | 1순위 |
| `server/routes/admin-routes.ts` | 3,575줄 | 🔴 매우 높음 | 2순위 |
| `shared/schema.ts` | 2,257줄 | 🟡 중간 | 3순위 (스키마는 분리 리스크 높음) |

## 2. 실제 사고 사례 — 2026-03-05 미션 라우트 분리

### 2.1 무엇이 일어났는가

미션 관련 코드를 리팩토링하면서 `admin/big-missions` 라우트가 `mission-routes.ts`로 이동됨.
그러나 서버가 재시작되지 않아 **4시간 이상 이전 코드(라우트 미포함)로 실행됨**.

결과:
1. **관리자 페이지**: 큰미션 0개 표시 (API가 HTML 반환)
2. **사용자 페이지**: `/mymissions` 빈 화면
3. **큰미션 수정**: 500 에러 (`req.body`에 relation 데이터 포함)

### 2.2 근본 원인 분석

| 원인 | 설명 |
|------|------|
| **서버 미재시작** | 새 라우트 파일이 코드에 있지만 런타임에 로드되지 않음 |
| **라우트 검증 미수행** | 분리 후 실제 API 응답 형태(JSON vs HTML) 확인 안 함 |
| **req.body 미필터링** | 프론트엔드가 보내는 relation 데이터(topics, hospital)가 DB update에 직접 전달 |
| **dev visibility 미고려** | superadmin에게도 dev 미션이 필터링됨 |

### 2.3 교훈

> ❗ **리팩토링은 코드를 옮기는 것이 아니라, 동작을 보존하는 것이다.**
> 
> 코드가 컴파일되어도 런타임에 제대로 동작하는지 반드시 검증해야 한다.
> 특히 Express 라우트는 "등록"과 "존재"가 다른 개념이다.

### 3.3 Step 1: 단일 도메인 분리

#### 핵심 원칙: **한 번에 하나의 도메인만 분리**


#### 코드 이동 방법: "복사 → 확인 → 삭제" (3단계)

```
1단계 - 복사: 분리할 코드를 새 파일에 복사 (원본은 그대로 유지)
2단계 - 확인: 새 파일에서 import 연결 후, 서버 재시작하여 동작 확인
3단계 - 삭제: 확인이 완료된 후에만 원본 파일에서 해당 코드 삭제
```

**절대 "잘라내기 → 붙여넣기"를 하지 마세요!** 복사 → 확인 → 삭제 순서를 지켜야 합니다.

## 4. 파일 유형별 분리 전략

### 4.1 백엔드 라우트 파일 (admin-routes.ts 3,575줄)

**분리 단위**: 도메인(기능 영역)별

```
admin-routes.ts (3,575줄)
├── admin-hospital-routes.ts    # 병원 관리 CRUD
├── admin-milestone-routes.ts   # 마일스톤 관리
├── admin-service-routes.ts     # 서비스 카탈로그
├── admin-persona-routes.ts     # 페르소나 관리
├── admin-concept-routes.ts     # 컨셉 관리
├── admin-system-routes.ts      # 시스템 설정
└── admin-routes.ts             # 나머지 + 라우트 통합 등록
```

**분리 순서**: 가장 독립적인(다른 라우트와 의존성이 적은) 도메인부터 시작.

**주의사항**:
- 각 파일에서 `requireAdminOrSuperAdmin` 미들웨어 import 필수
- 유틸 함수(`getErrorMessage`, `validateUserId` 등)는 공통 파일로 먼저 추출
- `server/routes.ts`의 등록 순서에서 기존과 동일한 위치에 새 라우트 삽입

### 4.2 프론트엔드 대형 컴포넌트 (MissionManagement.tsx 3,591줄)

**분리 단위**: UI 컴포넌트 + Custom Hook

```
components/admin/mission/
├── MissionManagement.tsx          # 메인 (탭 라우터 역할만)
├── ActionTypeManagement.tsx       # 액션 타입 관리 탭
├── MissionCategoryManagement.tsx  # 카테고리 관리 탭

├── SortableFolderSection.tsx      # 드래그 가능 폴더 섹션
└── hooks/
```

**분리 순서**: 
1. 독립 컴포넌트부터 (`SortableMissionRow`, `SortableFolderSection`)
2. 탭 컴포넌트 (`ActionTypeManagement`, `MissionCategoryManagement`)
3. Custom Hook 추출 (`useMissionData`, `useMissionMutations`)
4. 메인 파일 정리

**주의사항**:
- Props 타입(interface)을 먼저 공통 타입 파일로 추출
- 컴포넌트 내부에서만 사용하는 상태는 해당 컴포넌트로 이동
- 여러 컴포넌트가 공유하는 상태는 부모(MissionManagement)에 유지

### 4.3 스키마 파일 (schema.ts 2,257줄)

> ⚠️ **스키마 분리는 가장 위험도가 높은 작업입니다.**

Drizzle ORM의 relation 정의가 테이블 간 순환 참조를 가지고 있어,
잘못 분리하면 런타임 에러가 발생합니다.

**분리 단위**: 도메인 그룹

```
shared/schema/
├── index.ts           # 전체 re-export
├── users.ts           # users, userRoles, userNotificationSettings
├── hospitals.ts       # hospitals, hospitalMembers, hospitalCodes
├── missions.ts        # 미션 관련 모든 테이블
├── content.ts         # images, music, personas
├── milestones.ts      # milestones, milestoneApplications
└── relations.ts       # 모든 relation 정의 (한 곳에 모아야 함!)
```

**핵심 주의**: `relations.ts`는 반드시 하나의 파일에 모아야 합니다.
Drizzle ORM에서 relation 정의가 분산되면 순환 import 문제 발생.

**분리 순서**: 스키마 분리는 마지막에 진행. 다른 모든 리팩토링이 완료된 후에만.

#### 스키마 분리 예시 (순환 참조 방지 패턴)

```typescript
// ❌ 피해야 할 파일 구조 (순환 참조 발생)
// users.ts -> import { hospitals } from './hospitals'
// hospitals.ts -> import { users } from './users'

// ✅ 올바른 구조 (테이블 모듈과 관계 모듈 분리)
// 1. schema/users.ts (테이블 정의만 작성)
export const users = pgTable("users", { /* ... */ });

// 2. schema/hospitals.ts (테이블 정의만 작성)
export const hospitals = pgTable("hospitals", { /* ... */ });

// 3. schema/relations.ts (모든 관계 정의를 한 곳에 집중)
import { users } from "./users";
import { hospitals } from "./hospitals";

export const usersRelations = relations(users, ({ one }) => ({
  hospital: one(hospitals, { fields: [users.hospitalId], references: [hospitals.id] }),
}));
```

## 5. 자주 발생하는 문제 및 해결법

### 5.1 라우트 등록했는데 HTML 반환

**원인**: 라우트가 코드에 존재하지만 Express에 등록되지 않음
**원인 상세**:
- `server/routes.ts`(또는 메인 라우터 등록 파일)에서 새 라우트 파일을 import하지 않음
- `app.use()`로 등록하지 않음
- 서버를 재시작하지 않음

**해결**:
1. `server/routes.ts`에서 import 확인
2. `app.use('/api', newRouter)` 등록 확인
3. **서버 재시작**

### 5.2 req.body에 관계 데이터 포함으로 500 에러

**원인**: 프론트엔드에서 API 호출 시 relation 데이터(`topics`, `hospital` 등)가 함께 전송됨
**원인 상세**: `db.update().set(req.body)`처럼 body 전체를 DB에 전달하면,
실제 테이블 컬럼이 아닌 필드가 포함되어 에러 발생

**해결**: 화이트리스트 패턴 적용

```typescript
// ❌ 위험
const [result] = await db.update(table).set(req.body);

// ✅ 안전 (타입스크립트 규칙 준수: 강력한 타입 및 화이트리스트 적용)
type UpdateData = Partial<typeof table.$inferInsert>; // Drizzle 타입 활용

const allowedFields: (keyof UpdateData)[] = ['title', 'description', 'isActive'];
const updateData = {} as UpdateData;

for (const field of allowedFields) {
  if (req.body[field] !== undefined) {
    updateData[field] = req.body[field];
  }
}
const [result] = await db.update(table).set(updateData);
```

### 5.3 import 경로 깨짐

**원인**: 파일 위치를 옮기면서 상대 경로가 변경됨
**해결**: 
- `@/` 별칭(alias) 사용으로 상대 경로 의존 최소화
- 분리 후 `npx tsc --noEmit`으로 타입 체크

### 5.4 컴포넌트 분리 후 props 누락

**원인**: 기존에 부모 컴포넌트 스코프에서 직접 접근하던 변수를 props로 전달해야 하는데 누락
**해결**:
- interface를 먼저 정의하고, 모든 의존 변수를 props로 명시
- TypeScript 컴파일러가 누락을 잡아줌 → `noEmit` 체크 필수

### 5.5 서버 재시작해도 반영 안 됨

**원인**: `node.exe` 프로세스가 종료되지 않고 남아있음
**해결**:
```bash
# ⚠️ 주의: taskkill 명령은 실행 중인 "모든" node.exe 프로세스를 종료합니다. (다른 프로젝트 포함)
# 1순위: 터미널에서 Ctrl + C 로 정상 종료를 먼저 시도하세요.
# 2순위: Ctrl + C 로 종료가 안 되거나 좀비 프로세스가 남았을 때만 아래 명령 사용
taskkill /f /im node.exe
# 잠시 대기 후 재시작
Start-Sleep -Seconds 2
npm run dev
```


```
□ 새 파일 생성 완료

□ 새 파일에 필요한 import 추가 및 export 설정 완료
□ 기존 파일에서 새 파일을 import하여 교체
□ [백엔드] server/routes.ts(혹은 메인 라우터 파일)에 새 라우터 등록
□ ⚠️ [검증 필수] TS 컴파일 통과 및 서버 재시작 완료 (동작 확인 전제 조건)
□ ⚠️ [검증 필수] 새로 분리한 파트와 연동되는 기능/API 브라우저에서 정상 작동 확인
□ 정상 확인 완료 후, 기존 파일(원본)의 코드 삭제
```
