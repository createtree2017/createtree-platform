# 1. 개요

나의미션의 진행미션은 특정 문화센터 프로그램 ID가 아니라 `categoryId` 기준 슬롯이다. 같은 `categoryId`를 가진 문화센터 프로그램 중 사용자가 아무 회차의 프로그램 하나를 끝까지 완료하면 해당 진행미션을 완료로 본다.

기존 구현은 같은 카테고리의 세부미션 제출 중 하나라도 `approved`이면 진행미션을 완료로 계산했다. 이 때문에 문화센터 프로그램의 `신청` 세부미션만 승인되어도 나의미션이 완료되고, 사용자가 완료 선물을 신청할 수 있는 문제가 발생했다.

이번 보정은 큰미션 완료 계산을 공통 서비스로 분리하고, 사용자 API, 보상 신청 API, 관리자 검수 후 재계산, 문화센터 진행률 계산을 동일 기준으로 맞추는 작업이다.

# 2. 완료 기준

나의미션 진행미션 완료 조건:

- 진행미션은 `bigMissionTopics.categoryId`를 기준으로 판단한다.
- 같은 `categoryId`의 `themeMission` 중 하나라도 완료된 프로그램이 있으면 해당 진행미션 완료다.
- 프로그램 하나의 완료 조건은 다음과 같다.
  - `isActive=true`인 `subMissions`가 1개 이상 있다.
  - 해당 활성 세부미션 전체에 대해 같은 사용자의 `subMissionSubmissions.status='approved'` 제출이 존재한다.
- 비활성 세부미션은 분모와 분자에서 모두 제외한다.
- `신청` 세부미션 하나만 approved 된 상태는 다른 활성 세부미션이 남아 있으면 완료가 아니다.

# 3. 개발 범위

## 3.1 Backend

- 큰미션 완료 계산 공통 서비스 추가
  - 사용자별 완료 category map 계산
  - 큰미션별 topic 평가
  - `user_big_mission_progress` 재계산
- 사용자 API 교체
  - `GET /api/big-missions`
  - `GET /api/big-missions/:id`
  - `POST /api/big-missions/:id/apply-reward`
- 관리자 검수 후 재계산 교체
  - `admin.mission.review.service.ts`
- 문화센터 진행률 계산 보정
  - 분모: 활성 세부미션
  - 분자: 활성 세부미션의 approved 제출

## 3.2 DB/API

- 보상 신청 운영취소 이력 컬럼 추가
  - `rewardStatus='cancelled'`
  - `rewardCancelledAt`
  - `rewardCancelledBy`
  - `rewardCancelReason`
- 관리자 pending 보상 신청 운영취소 API 추가
- approved 지급완료 건은 자동취소하지 않고 수동 검토 대상으로 남긴다.

## 3.3 Admin UI

- 보상 신청 관리에 `cancelled` 상태 표시와 필터 추가
- pending 신청에 운영취소 버튼 추가
- 취소일/취소사유 표시

## 3.4 운영 데이터 dry-run

- 기존 `user_big_mission_progress` 기준 완료 상태와 새 기준 완료 상태를 비교한다.
- pending/approved 보상 신청 중 새 기준으로 미완료인 취소/수동검토 대상을 출력한다.
- 실제 UPDATE/DELETE는 하지 않는다.
