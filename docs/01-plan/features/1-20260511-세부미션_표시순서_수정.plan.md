# 세부미션 표시 순서 수정 Plan

작성일: 2026-05-11

## 작업 배경

관리자 페이지의 `미션 시스템 관리 > 주제 미션 > 세부미션 관리`에서 드래그앤드롭으로 세부미션 순서를 바꾸면 `잘못된 요청 형식입니다` 오류가 발생한다.
조사 결과 프론트엔드는 `subMissionIds` 숫자 배열을 보내지만, 서버는 `subMissionOrders: [{ id, order }]` 형식을 기대하고 있어 API 계약이 어긋나 있었다.

추가로 세부미션 수정 화면에는 표시 순서(`order`) 입력이 없고, 순차 잠금용 `순차 등급(sequentialLevel)`만 있어 운영자가 두 값을 혼동할 여지가 있다.

## 구현 방침

- `order`는 화면 표시 순서로 유지하고, `sequentialLevel`은 순차 잠금 등급으로 유지한다.
- 드래그 완료 시 프론트에서 새 배열의 인덱스를 `order` 값으로 만들어 `subMissionOrders`를 전송한다.
- 서버는 `subMissionOrders`를 검증하고, 순서 업데이트를 한 작업 단위로 처리한다.
- 기존 클라이언트 호환을 위해 필요 시 `subMissionIds`도 서버에서 변환 가능한 형태로 허용한다.
- 세부미션 부분 업데이트 시 전달되지 않은 날짜/순차등급이 임의로 초기화되지 않도록 서비스 정규화 로직을 보강한다.
- 세부미션 수정 모달의 `표시 순서` 설명에는 “순차 등급과 다름”을 명시한다.
- `신청` 액션 타입 세부미션은 행사 신청 영역에 고정 표시되는 접수/선정용 미션으로 취급한다.
- `신청` 타입 저장 시 서버에서 `order=0`, `requireReview=false`로 정규화해 프론트 우회나 기존 데이터 편집 상황에서도 정책을 유지한다.
- 관리자 세부미션 폼에서는 `신청` 타입 선택 시 표시순서 입력과 검수 필요 스위치를 비활성화하고, 선착순 신청 자동승인 정책 안내를 표시한다.
- 주제미션 폴더 목록은 서버 응답이 실제로 변경된 경우에만 로컬 상태와 동기화하여 `useEffect -> setLocalFolders -> rerender` 반복 가능성을 차단한다.
- 세부미션 목록에서는 액션 타입 관계 데이터가 `신청`일 때 `검수 필요` 대신 `신청 정책 적용` 배지를 표시하고, 관계 데이터가 없으면 기존 `requireReview` 표시를 유지한다.
- 문화센터 리스트 카드의 대기자 수는 `mission.isFirstCome && waitlistCount > 0` 조건에서만 표시한다.
- 문화센터 리스트 카드의 참여 방식은 `누구나 참여`, `선정`, `선착순` 뱃지로 구분한다. 정원 있는 미션은 모집인원 행 안에 `모집인원 | [선정/선착순]` 형태로 표시하고, 정원 없는 미션은 `누구나 참여` 뱃지를 단독 표시한다.
- 리스트의 모집 진행바는 기존처럼 `currentApplicants / capacity` 기준만 사용하고, 대기자 수는 진행률에 포함하지 않는다.
- 리스트 API의 선착순 대기자 집계는 신청 세부미션의 `waitlist` 제출을 `count(DISTINCT userId)`로 계산해 중복 제출 데이터에 안전하게 처리한다.

## 검증 계획

- `npm run build`
- 가능하면 `npm run check` 실행 후 기존 타입 오류와 신규 오류를 구분한다.
- 수동 테스트:
  1. 세부미션 관리 모달에서 드래그앤드롭 후 오류 없이 저장되는지 확인
  2. 세부미션 수정 모달에서 `표시 순서`를 변경하고 저장 후 목록 순서가 바뀌는지 확인
  3. `순차 등급` 값이 표시 순서 변경과 별개로 유지되는지 확인
  4. 액션 타입을 `신청`으로 선택하면 표시순서와 검수 필요가 비활성화되고 정책 안내가 표시되는지 확인
  5. 액션 타입을 일반 타입으로 바꾸면 표시순서와 검수 필요를 기존처럼 사용할 수 있는지 확인
  6. 서버 API가 `신청` 타입 생성/수정 요청의 `order`, `requireReview` 입력값을 정책값으로 정규화하는지 확인
  7. 관리자 문화센터 주제 미션 탭 진입 시 `Maximum update depth exceeded` 콘솔 경고가 반복되지 않는지 확인
  8. 세부미션 관리 모달에서 `신청` 세부미션은 정책 배지, 일반 제출 세부미션은 기존 검수 배지가 표시되는지 확인
  9. 문화센터 리스트에서 선착순 대기자 수가 `1 / 1명 (대기 1명)` 형태로 표시되는지 확인
  10. 같은 미션 상세 페이지의 기존 `(대기 N명)` 표시가 유지되는지 확인
  11. 선착순이 아닌 미션에는 대기자 수가 표시되지 않는지 확인
