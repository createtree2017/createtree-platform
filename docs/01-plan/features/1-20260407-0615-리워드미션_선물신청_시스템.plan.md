# 1. 개요

사용자가 나의미션(큰미션)의 진행률을 100% 달성했을 때 관리자가 등록한 보상 종류별 수량을 직접 선택하여 선물 신청할 수 있도록 한다. 관리자는 기존처럼 보상 설정에서 `이미지 + 보상 이름`을 직접 등록하며, 코드는 상품명을 해석하거나 자동 조합하지 않는다.

기존 2026-04-07 계획은 `rewardStatus` 기반 신청/승인 흐름을 구축하는 범위였다. 2026-06-15 확장 범위는 신청 시 선택한 보상을 스냅샷으로 저장하고 관리자 화면에서 확인하는 기능이다.

2026-06-15 추가 확장 범위는 단일 보상 선택을 `보상 종류별 수량 선택`으로 바꾸는 것이다. 관리자는 보상 조합을 모두 등록하지 않고 보상 종류만 등록하며, 사용자는 각 보상별 수량을 골라 큰미션별 총 선택 가능 수량을 맞춰 신청한다. 이후 배송 받을 주소와 사용자 메모를 함께 신청하도록 확장한다. 추가로 관리자가 아직 지급완료 처리하지 않은 `pending` 상태에서는 사용자가 신청 내용을 수정할 수 있게 한다. 모든 사용자 UI는 모바일 화면을 1차 기준으로 설계한다.

# 2. 핵심 요구사항

- 보상 상품명은 코드에 고정하지 않는다.
- 자동 분류, 자동 조합, 자동 추천은 구현하지 않는다.
- 관리자는 큰미션 보상 설정에서 `이미지 + 보상 이름`을 직접 등록한다.
- 사용자는 미션 완료 후 등록된 보상 종류별 수량을 선택한다.
- 기존 단일 선택 API 요청은 backward compatibility 목적으로만 유지한다.
- 큰미션별 `총 선택 가능 수량`을 설정한다.
- 기존 미션의 총 선택 가능 수량 기본값은 1개로 유지한다.
- 신청 시 선택한 보상 정보를 저장한다.
- 신청 시 배송 받을 주소를 필수로 저장한다.
- 신청 시 사용자 메모를 선택값으로 저장한다.
- 신청 대기 상태에서는 사용자가 보상 수량, 배송 주소, 메모를 수정할 수 있다.
- 지급 완료 상태에서는 사용자가 신청 내용을 수정할 수 없다.
- 관리자가 이후 보상 목록을 수정해도 기존 신청자의 선택 내역은 보존한다.
- 기존 신청 건처럼 선택 보상이 없는 데이터는 "선택 정보 없음"으로 안전하게 표시한다.
- 기존 신청 건처럼 배송 주소/메모가 없는 데이터는 "배송 정보 없음", "메모 없음"으로 안전하게 표시한다.

# 3. 데이터베이스 계획

작업 파일: `shared/schema.ts`, `db/migrations/`

기존 `userBigMissionProgress` 테이블의 리워드 상태 컬럼은 유지한다.

- `rewardStatus`: `not_eligible` / `can_apply` / `pending` / `approved`
- `rewardAppliedAt`: 사용자 신청 일시
- `rewardProcessedAt`: 관리자 처리 일시

이번 확장에서 추가할 컬럼:

- `selectedRewardItem`: `jsonb`, DB 컬럼명 `selected_reward_item`
- `rewardSelectionLimit`: `integer`, DB 컬럼명 `reward_selection_limit`, 기본값 1
- `rewardShippingAddress`: `text`, DB 컬럼명 `reward_shipping_address`
- `rewardMemo`: `text`, DB 컬럼명 `reward_memo`
- 저장 스냅샷 예시:

```json
{
  "imageUrl": "/uploads/example.webp",
  "description": "물티슈 1박스",
  "selectedIndex": 0
}
```

수량 선택 저장 스냅샷 예시:

```json
[
  {
    "imageUrl": "/uploads/wipes.webp",
    "description": "물티슈 1박스",
    "selectedIndex": 0,
    "quantity": 1
  },
  {
    "imageUrl": "/uploads/diaper.webp",
    "description": "기저귀 1팩",
    "selectedIndex": 1,
    "quantity": 2
  }
]
```

스냅샷으로 저장하는 이유는 신청 이후 관리자가 `giftItems`를 수정하거나 삭제해도 신청 당시 사용자가 선택한 보상을 보존하기 위해서다.

# 4. 개발 범위

## 4.1 Backend API

- `POST /api/big-missions/:id/apply-reward`
  - 요청 body에서 `selectedRewardIndex`를 받는다.
  - 2026-06-15 수량 선택 확장 후에는 `selectedRewards: [{ selectedIndex, quantity }]`를 받는다.
  - 기존 단일 선택 호환을 위해 `selectedRewardIndex` 요청도 수량 1개로 처리한다.
  - 미션 100% 완료 여부를 검증한다.
  - 중복 신청 또는 지급 완료 상태를 차단한다.
  - 선택한 index가 현재 큰미션의 보상 목록 안에 있는지 검증한다.
  - 수량이 0 이상 정수인지 검증한다.
  - 수량 합계가 큰미션의 `rewardSelectionLimit`와 정확히 같은지 검증한다.
  - 배송 받을 주소는 trim 후 빈 값이면 거부한다.
  - 사용자 메모는 선택값으로 받으며, 빈 값은 `null`로 저장한다.
  - 검증된 보상과 수량을 `selectedRewardItem` 배열 스냅샷으로 저장한다.
  - 배송 주소와 메모를 함께 저장한다.
  - 기존 진행 row가 `pending` 상태이면 동일 검증을 거쳐 신청 내용을 수정 저장한다.
  - 수정 저장 시 `rewardAppliedAt`은 최초 신청일로 유지하고 `selectedRewardItem`, `rewardShippingAddress`, `rewardMemo`, `updatedAt`만 갱신한다.
  - 기존 진행 row가 `approved` 상태이면 서버에서 수정 또는 재신청을 거부한다.

- `GET /api/big-missions/:id`
  - 사용자의 `selectedRewardItem`을 내려준다.
  - 신청 완료/지급 완료 상태 표시를 위해 `rewardShippingAddress`, `rewardMemo`를 내려준다.

- `GET /api/admin/big-missions/rewards/applications`
  - 신청 목록에 `selectedRewardItem`을 포함한다.
  - 신청 목록에 `rewardShippingAddress`, `rewardMemo`를 포함한다.

## 4.2 사용자 화면

작업 파일: `client/src/pages/mymission-detail.tsx`

- 미션 완료 전: 기존처럼 잠긴 보상 목록을 표시한다.
- 미션 완료 후 신청 전: 보상 목록을 모바일 터치용 `+ / -` 수량 선택 카드로 표시한다.
- 사용자가 총 선택 가능 수량을 정확히 맞춰야 `선물 신청하기` 버튼을 활성화한다.
- 신청 완료/지급 완료 상태에서는 선택한 보상을 표시한다.
- 라디오 선택 UI는 제거하고 모바일 터치용 `+ / -` 수량 선택 UI를 표시한다.
- 선택 합계와 총 선택 가능 수량을 명확히 표시하고, 합계가 정확히 맞을 때만 신청 버튼을 활성화한다.
- 수량 선택 UI 아래에 배송 받을 주소 입력란과 메모 입력란을 표시한다.
- 주소가 비어 있으면 신청 버튼을 비활성화하고, 메모는 선택값으로 둔다.
- 신청 완료/지급 완료 상태에서는 선택 보상, 수량, 배송 주소, 메모를 함께 표시한다.
- 신청 대기 상태에서는 `신청 내용 수정` 버튼을 표시한다.
- 수정 모드에서는 기존 신청값을 수량 선택 UI와 주소/메모 입력란에 채워서 표시한다.
- 수정 모드에서는 수량 합계와 배송 주소 조건이 만족될 때만 `수정 저장하기` 버튼을 활성화한다.
- 지급 완료 상태에서는 수정 버튼을 표시하지 않는다.
