# FCM 디바이스 토큰 백엔드 연동 계획서 (Plan)

## 1. 개요
모바일 앱(APK)의 핵심 기능인 **푸시 알림(FCM)**을 구축하기 위한 백엔드 연동 작업입니다. 
앱이 실행될 때 기기에서 발급받은 FCM 토큰을 서버로 전송하고, 서버는 이를 Drizzle ORM을 통해 데이터베이스에 저장하여 향후 알림 발송 시 사용하게 됩니다.

현재 "멤버십 전용 앱 자체 배포 (1단계)" 전략에 맞춰, 우선적으로 안드로이드(APK) 환경에서의 토큰 수집 및 저장을 목표로 합니다.

## 2. 작업 상세 내용

### 2.1. 데이터베이스 스키마 설계 (`shared/schema.ts`)
사용자는 여러 대의 기기(스마트폰, 태블릿 등)를 사용할 수 있으므로, 단일 컬럼이 아닌 별도의 디바이스 토큰 테이블을 생성하는 것이 좋습니다.

*   **새로운 테이블 추가:** `user_devices` (또는 `fcm_tokens`)
*   **컬럼 구성:**
    *   `id`: 기본키 (serial)
    *   `userId`: 사용자 ID (users 테이블 참조)
    *   `deviceToken`: FCM 고유 토큰 값 (text, unique)
    *   `deviceType`: 기기 종류 (예: 'android', 'ios', 'web') - 기본값 'android'
    *   `isActive`: 토큰 유효 상태 (boolean) - 기본값 true
    *   `lastUsedAt`: 마지막으로 토큰이 갱신/사용된 날짜 (timestamp)
    *   `createdAt`: 최초 등록일 (timestamp)

### 2.2. 백엔드 API 개발 (`server/routes.ts` 또는 `server/services/users.ts`)
기기에서 발급된 토큰을 서버로 전달받아 DB에 저장/업데이트(Upsert)하는 API 엔드포인트를 구축합니다.

*   **엔드포인트:** `POST /api/users/device-token`
*   **요청 데이터 (Request Body):**
    ```json
    {
      "token": "fcm_device_token_string_here",
      "deviceType": "android"
    }
    ```
*   **처리 로직:**
    1.  사용자 인증(로그인 상태) 확인
    2.  전달받은 토큰이 이미 DB에 있는지 확인
    3.  없다면 새로 추가 (Insert)
    4.  있지만 다른 사용자의 것이거나 비활성화 상태라면 현재 사용자로 소유자 변경 및 활성화 (Update)
    5.  `lastUsedAt` 시간 갱신

### 2.3. 프론트엔드 연동 준비 (코드 점검)
*   **작업 내용:** 앞서 프론트엔드 작업에 준비해 둔 "푸시 권한 획득 훅(Hook)"에서 토큰을 정상적으로 받아올 때, 새로 만든 `POST /api/users/device-token` API를 호출하도록 쿼리(fetch/axios 로직)를 연결합니다.

## 3. 기대 효과
*   앱 설치 후 로그인한 사용자의 고유 푸시 식별자가 서버와 동기화됩니다.
*   관리자나 시스템이 이벤트, 미션 승인 등의 알림을 특정 사용자의 기기로 정확히 발송할 수 있는 인프라가 마련됩니다.

## 4. 검토 요청 사항
*   **토큰 테이블 구조:** 기기 관리 목적으로 위에서 제안한 `user_devices` 테이블 구조로 추가하는 방향이 적절할까요?
*   승인되시면 바로 Database 모델링(Design) 및 구현 코드 작성을 진행하겠습니다.
