# 앱 푸시 (App Push) 시스템 구축 계획서

> **작성일:** 2026-03-26
> **프로젝트:** createTree AI 콘텐츠 플랫폼
> **아키텍처:** React + Capacitor + Express + Firebase Cloud Messaging (FCM v1)

---

## 1. 아키텍처 개요 및 핵심 원칙
- **FCM v1 API 사용:** Legacy API 폐기(2024.06)에 따른 `firebase-admin` (v10+) 기반 발송
- **안전한 발송 (Fault Tolerance):** 500개 단위 청크 분할 및 `Promise.allSettled()`를 통한 에러 격리 처리
- **DB와 외부 API의 상태 분리:** 푸시 발송(`FCM API`)을 DB 트랜잭션으로 묶지 않고 단계별(pending -> completed) 상태 업데이트 적용
- **1계정 N토큰 및 토큰 생명주기 관리:** 미사용/만료 토큰의 능동적 파기(`UNREGISTERED` 에러 핸들링)
- **보안:** Firebase Admin Service Account Key는 `.env`에 base64 인코딩하여 안전하게 주입

## 2. DB 스키마 설계 (PostgreSQL / Drizzle)

### A. `device_tokens` (기기 토큰 관리)
- `id` (serial, PK)
- `user_id` (integer, FK)
- `token` (varchar, Unique) : FCM에서 발급된 기기 고유 토큰
- `platform` (varchar) : 'android' | 'ios' | 'web'
- `is_active` (boolean, default: true) : 유효 토큰 및 알림 수신 상태 제어 필터 레이어
- `last_used_at` (timestamp) : 정기적인(예: 90일) 미사용 토큰 정리를 위한 타임스탬프
- `created_at`, `updated_at`

### B. `notifications` (유저 알림장 내역)
- `id` (serial, PK)
- `user_id` (integer, FK)
- `type` (varchar) : 'mission_approved', 'grade_up', 'system_notice', 'gift' 등 (분류용 Enum)
- `title` (varchar), `body` (text)
- `is_read` (boolean, default: false) : 읽음 여부
- `read_at` (timestamp, nullable) : 읽은 시간
- `action_url` (varchar, nullable) : 푸시 클릭 시 이동할 딥링크 (예: `/mymission/123`)
- `image_url` (varchar, nullable) : 리치 푸시용 썸네일
- `created_at`
- **데이터 보존 정책:** 90일이 지난 오래된 알림은 서버 Cron Job 또는 배치를 통해 자동 영구 삭제 처리하여 DB 용량 최적화.

### C. `push_delivery_logs` (관리자 발송 이력)
- `id` (serial, PK)
- `admin_id` (integer, FK, nullable) : 수동 발송 시 관리자 ID, 시스템 자동 발송 시엔 null
- `target_type` (varchar) : 'all' | 'hospital' | 'specific_user'
- `target_query` (jsonb, nullable) : 구조화된 타겟팅 조건 (예: `{"hospital_id": 5}`, `{"user_ids": [123, 456]}`)
- `title`, `body`
- `status` (varchar) : 'pending' | 'completed' | 'failed' (트랜잭션 분리용)
- `success_count` (integer), `failure_count` (integer)
- `created_at`, `completed_at`

### D. `users` (기존 테이블 확장)
- `is_system_push_agreed` (boolean, default: true) : 정보통신망법 허용 시스템성 알림(미션승인 등) 수신 동의
- `is_marketing_push_agreed` (boolean, default: false) : 정보통신망법 Opt-in 대응 마케팅/이벤트 알림 수신 동의

## 3. 백엔드 (Express) 구현 로직

### A. 배치 (청크) 발송 및 Promise.allSettled
- **대상 인원 수와 관계 없이 항상** 발송 배열을 최대 500개 단위로 엄격히 쪼개어 청크 처리 (`chunkArray` 유틸)
- `Promise.all()` 대신 **`Promise.allSettled()`**를 사용하여 중간 배열 전송 실패가 전체 시스템을 멈추지 않도록 보호.

### B. 좀비 토큰 에러 핸들링
- FCM 전송 후 발송 결과(responses)를 순회 검사
- `UNREGISTERED` 또는 `SENDER_ID_MISMATCH` 에러 시: 기기에서 앱 삭제가 확실하므로 즉시 토큰을 DB에서 `DELETE` 처리함과 동시에, FCM 구글 서버의 **`unsubscribeFromTopic`도 연달아 호출**하여 구독 쓰레기 값을 청소.
- `INVALID_ARGUMENT` 에러 시: 메시지 페이로드 오류일 수 있으므로 삭제 금지(로그만 기록).

### C. FCM 토픽(Topic) 기반 발송 (확장성)
- 토큰 갱신(Upsert) API 호출 시, 서버(`firebase-admin`)에서 해당 기기를 `all_users` 및 `hospital_{병원ID}` 토픽에 자동 구독/해지 처리.
- 전체 공지 및 병원 공지 발송 시 DB 조회 없이 다이렉트 브로드캐스트 수행하여 성능 최적화.

### D. Rate Limiting (API 남용 방지)
- 관리자 푸시 발송 엔드포인트 속도 제한(Rate Limiting) 명시적 설계:
  - **전체 발송 API:** 분당 최대 1회
  - **병원별/개별 발송 API:** 분당 최대 5회
- 중복 요청을 방어하여 "버튼 연타"로 인한 대참사 차단

## 4. 클라이언트 (React + Capacitor) 로직

### A. 토큰 자동 갱신 및 DB 동기화
- `App.tsx` 최상단에서 Capacitor `addListener('registration')` 훅 실행하여 발급받은 토큰을 서버 `upsert` API로 전송, 항상 최신 상태 유지
- **토큰 에러 방어:** `addListener('registrationError', (err) => {...})` 훅을 병행 배치하여, 네트워크/설정 오류로 기기가 토큰을 발급받지 못하는 치명적 상태가 되면 로그를 남기거나 재시도(Retry) 처리.

### B. 딥링크 (Deep Link) 및 Badge, 읽음 처리
- `pushNotificationActionPerformed` 리스너에서 `action_url` 추출 후 React Router `useNavigate` 이동 (포그라운드/백그라운드 지원)
- **앱 완전 종료(Terminated) 상태 딥링크 방어:** 기기가 재부팅되거나 앱이 완전히 꺼진 상태에서 푸시로 진입할 경우를 캐치하기 위해, `App.tsx` 마운트 시 `PushNotifications.getDeliveredNotifications()` 나 Capacitor `appUrlOpen`을 초기 검증하는 로직 이중 구현.
- **읽음 처리 및 뱃지 초기화:** 사용자가 [🔔 알림장] 페이지에 진입하는 순간 Capacitor `Badge.clear()`를 호출하고, 알림을 "일괄 읽음 처리(is_read=true)" 하는 카카오/네이버 방식을 채택하여 서버 부하 및 UX 간소화.

### C. 포그라운드(Foreground) 알림 표시 및 이중 알림 방어
- 앱이 켜져 있을 때 수신되는 `pushNotificationReceived` 리스너 감지 시 동작
- **플랫폼별 이중 알림(Double Notification) 방어:** iOS는 `presentationOptions: ["alert"]` 설정 시 네이티브 OS가 상단 배너를 무조건 띄우므로 동일 알림이 2개 뜨지 않도록 `toast` 생략. 안드로이드(또는 Web) 환경에서만 단일화된 **`use-toast` 시스템**을 호출하여 인앱(In-app) 알림을 부드럽게 띄움.

## 5. 안드로이드 / iOS 네이티브(Native) 필수 설정

### A. Android: 알림 채널 (Notification Channel) 등록 [중요 🔴]
- Android 8.0+ 필수 규격으로 무음/알림 누락 방지
- **Capacitor 초기에 채널 ID를 명시적으로 등록:**
  - `mission_alerts` (중요도: High - 화면 팝업/소리)
  - `system_notices` (중요도: Default)

### B. iOS: 포그라운드 presentationOptions 설정
- `capacitor.config.ts` 내 Plugins 설정:
  ```json
  "PushNotifications": {
    "presentationOptions": ["badge", "sound", "alert"]
  }
  ```
- 이 세팅이 누락되면 iOS는 포그라운드 상태에서 네이티브 배너를 강제 무시함.
- **주의:** iOS 네이티브 빌드는 Apple Developer 계정($99) 및 Mac 장비 기반의 Xcode가 100% 필수임.

## 6. 어드민 발송 UI 및 가이드

- **2단계 확인 모달:** 전체 계정 발송 시 `총 N명에게 발송합니다. 진행하시겠습니까?` 브레이크 다이얼로그 필수
- **발송 이력(Log) 조회 대시보드:** 관리자가 그동안 보냈던 이력을 볼 수 있는 테이블 관리자 페이지 추가 구현 (일시, 발송자, 대상영역, 내용, 성공/실패 수)
- 타겟팅 UI 옵션: 전체 발송 / 병원별 발송 / 개별 회원 검색 발송

## 7. 시스템 자동 발송 트리거 및 흐름도

**동시 실행(X) -> 순차 기반 설계(O):** 딥링크와 완벽한 연동을 위해 반드시 `1) DB 알림 저장(notification_id 발급)` -> `2) Payload에 notification_id 탑재하여 FCM 전송` -> `3) 클릭 시 notification_id 기준으로 API 연동` 순서로 설계합니다.
- **[중요] 안드로이드 Payload 매핑:** 백엔드 발송 로직에서 FCM Payload 객체 구성 시 `android.notification.channelId = "mission_alerts"` 와 같이 명시적으로 채널 ID를 맵핑해야만 안드로이드의 `Notification Channel`이 정상 작동합니다.

백엔드 로직 처리 시 다음 이벤트가 성공하면 위 순서가 자동 실행됩니다:
1. **미션 승인 완료:** "[승인] 회원님의 미션이 성공적으로 승인되었습니다." (action: `/mymission-detail`)
2. **미션 반려 (사유포함):** "[반려] 미션이 반려되었습니다. 사유를 확인해주세요." (action: `/mymission-detail`)
3. **포인트 지급/선물 발송:** "🎁 축하합니다! 리워드가 지급되었습니다." (action: `/mypage`)
4. **신규 가입 병원 소속 승인:** "가입이 승인되었습니다. 앱 활동을 시작해보세요!" (action: `/`)
5. **등급 갱신 (승급):** "🎉 회원 등급이 올라갔습니다! 새로운 혜택을 확인하세요." (action: `/mypage`)

## 8. 실제 개발 진행 (Step)

- **Step 1:** DB 스키마 생성 및 마이그레이션 (`device_tokens`, `notifications`, `push_delivery_logs`, `is_system_push_agreed`, `is_marketing_push_agreed`)
- **Step 2:** Express 백엔드 발송/수신 API 엔진, Rate Limit, Error(좀비토큰) 처리 로직 개발
- **Step 3:** React 클라이언트 Capacitor 연동 (권한 요청, 토큰 Upsert, 포그라운드 Toast, Badge) *— 토큰을 먼저 받아야 테스트 가능*
- **Step 4:** 네이티브 환경 설정 (안드로이드 `google-services.json` 및 채널 / iOS 설정 파일)
- **Step 5:** 시스템 자동 발송 트리거 연동 및 어드민 관리자 기능 개발
