## 2. 아키텍처 개요 및 핵심 원칙

### A. FCM v1 API 사용
- Legacy API 폐기(2024.06)에 따른 `firebase-admin` (v10+) 기반 발송
- Service Account Key는 `.env`에 `FIREBASE_SERVICE_ACCOUNT_BASE64`로 base64 인코딩하여 안전 주입

### B. 안전한 발송 (Fault Tolerance)
- 500개 단위 청크 분할 + `Promise.allSettled()`를 통한 에러 격리
- DB 트랜잭션과 FCM API를 분리: `pending → completed` 상태 전이

### C. 1계정 N토큰 + 토큰 생명주기 관리
- 한 유저가 여러 기기 로그인 시 모든 기기에 발송 (기존 `user_devices` 테이블이 이미 1:N 지원)
- 발송 실패 시 `UNREGISTERED` 에러로 좀비 토큰 자동 정리

### D. FCM 토픽(Topic) 기반 발송 (확장성)
- 토큰 등록 시 `all_users`, `hospital_{ID}` 토픽에 자동 구독
- 전체/병원별 공지 발송 시 DB 조회 없이 토픽으로 다이렉트 브로드캐스트

### E. 순차 기반 설계 (딥링크 완벽 연동)
```
1) DB 알림 저장 (notification_id 발급)
   → 2) Payload에 notification_id 탑재하여 FCM 전송
      → 3) 클릭 시 notification_id 기준으로 읽음 처리 + 딥링크 이동
```

### C. `push_delivery_logs` 테이블 — [신규 생성]

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| `id` | serial, PK | |
| `admin_id` | integer, FK (nullable) | 수동 발송 시 관리자 ID, 시스템 자동 발송 시 null |
| `trigger_type` | varchar | `'manual'` \| `'mission_approved'` \| `'mission_rejected'` \| `'grade_up'` \| `'gift'` \| `'system'` — admin_id가 null인 자동 발송의 원인을 구분하기 위한 필수 컬럼 |
| `target_type` | varchar | `'all'` \| `'hospital'` \| `'specific_user'` |
| `target_query` | jsonb (nullable) | 타겟팅 조건 (예: `{"hospital_id": 5}`, `{"user_ids": [123]}`) |

| `body` | text | 발송 본문 |
| `status` | varchar | `'pending'` \| `'completed'` \| `'failed'` |
| `success_count` | integer | 성공 건수 |
| `failure_count` | integer | 실패 건수 |
| `created_at` | timestamp | 발송 요청 시각 |
| `completed_at` | timestamp (nullable) | 발송 완료 시각 |

> 💡 **왜 `trigger_type`이 필요한가?** `admin_id=null`인 자동 발송 이력에서 "이 푸시가 왜 나갔지?"를 추적할 수 있어야 합니다. 예: 미션 승인인지, 등급 업인지, 선물 지급인지 구분 가능.

이미 필요한 컬럼이 모두 있음 (`userId`, `deviceToken`, `deviceType`, `isActive`, `lastUsedAt`)

기존 테이블 그대로 사용. `pushNotifications` 컬럼(1081줄)이 이미 있으므로 이것과 `users.is_system_push_agreed`의 역할 분리:

## 5. 각 신규 파일 상세 설계

### A. `server/utils/firebase-admin.ts` — SDK 초기화

```typescript
// firebase-admin SDK 싱글턴 초기화
// .env의 FIREBASE_SERVICE_ACCOUNT_BASE64를 디코딩하여 인증 정보 생성
// 앱 전체에서 getFirebaseAdmin()으로 호출

import admin from 'firebase-admin';

export function getFirebaseAdmin(): admin.app.App | null {
  // 이미 초기화된 앱이 있으면 재사용 (서버 시작 직후 동시 요청 방어)
  try {
    return admin.app();
  } catch (e) {
    // 아직 초기화 안 됨 → 아래에서 초기화 진행
  }

  const base64 = process.env.FIREBASE_SERVICE_ACCOUNT_BASE64;
  if (!base64) {
    console.warn('[Firebase] FIREBASE_SERVICE_ACCOUNT_BASE64 not set. Push disabled.');
    return null;
  }

  try {
    const serviceAccount = JSON.parse(Buffer.from(base64, 'base64').toString());
    return admin.initializeApp({ credential: admin.credential.cert(serviceAccount) });
  } catch (error) {
    console.error('[Firebase] Initialization failed:', error);
    return null;
  }
}
```

> 💡 **왜 `admin.app()` try-catch 패턴인가?** 이전 방식(`let initialized` 플래그)은 서버 시작 직후 동시 요청이 몰릴 때 `initializeApp()`이 두 번 호출될 이론적 위험이 있었습니다. `admin.app()`은 firebase-admin이 내부적으로 관리하는 싱글턴을 반환하므로 더 안전합니다.

### B. `server/services/push/push.service.ts` — 발송 핵심

핵심 역할: **토큰 배열을 받아 FCM으로 실제 발송하고, 결과를 반환**

```
주요 함수:
├── sendToUser(userId, payload)       # 특정 유저의 모든 활성 기기에 발송
├── sendToUsers(userIds, payload)     # 여러 유저에게 배치 발송
├── sendToTopic(topic, payload)       # 토픽(all_users, hospital_X) 브로드캐스트
├── sendBatch(tokens[], payload)      # 500개 청크 분할 + Promise.allSettled
└── handleFailedTokens(results)       # UNREGISTERED 에러 시 토큰 비활성화
```

**`sendToUser()` 토큰 0개 처리 (중요):**
```
sendToUser(userId, payload) {
  1. getActiveTokensByUserId(userId)로 활성 토큰 배열 조회
  2. if (tokens.length === 0) → 로그만 남기고 조용히 종료 (no-op)
     - 앱 미설치, 푸시 거부, 또는 웹 전용 사용자일 수 있음
     - DB 알림(notifications)은 이미 저장되었으므로 앱 내 알림함에서는 확인 가능
     - 에러를 throw하지 않음 (알림 생성 흐름을 방해하면 안 됨)
  3. if (tokens.length > 0) → sendBatch(tokens, payload) 호출
}
```

**청크 발송 로직 (핵심):**
```
1. tokens 배열을 500개씩 잘라 chunks[] 생성
2. 각 chunk에 대해 admin.messaging().sendEachForMulticast(message) 호출
3. Promise.allSettled()로 실행 → 한 chunk 실패해도 나머지 계속 진행
4. 각 response 순회:
   - success → 성공 카운트 증가
   - error.code === 'UNREGISTERED' → handleFailedTokens로 토큰 삭제
   - error.code === 'INVALID_ARGUMENT' → 로그만 기록 (삭제 금지)
5. 최종 {successCount, failureCount} 반환
6. push_delivery_logs에 발송 결과 INSERT (아래 상세)
```

**❗ `push_delivery_logs` INSERT 책임 — 이 서비스가 통합 처리:**

관리자 수동 발송이든 시스템 자동 발송이든, 최종적으로 `push.service`를 거치므로 **이 레이어에서 `push_delivery_logs`에 INSERT**합니다.
```
호출 체인:

[관리자 수동]: push.controller.ts → push.service.sendToUsers() → 발송 → logs INSERT (trigger_type: 'manual')
[시스템 자동]: notifications.ts → push.service.sendToUser()  → 발송 → logs INSERT (trigger_type: 'mission_approved' 등)

두 경우 모두 push.service 내부에서:
  1. FCM API 호출
  2. {successCount, failureCount} 집계
  3. push_delivery_logs INSERT (trigger_type, target_type, 결과 포함)
의 3단계를 순차 실행합니다.
```

> 💡 `trigger_type`은 `sendToUser()` / `sendToUsers()` 호출 시 매개변수로 전달받습니다.
> 예: `pushService.sendToUser(userId, payload, { triggerType: 'mission_approved', targetType: 'specific_user' })`

### C. `server/services/push/push.token.service.ts` — 토큰 관리

```
주요 함수:
├── upsertDeviceToken(userId, token, platform)  # 기존 profile.ts에서 이동 + 토픽 구독 추가
├── subscribeToTopics(token, userId)            # all_users + hospital_{ID} 토픽 자동 구독
├── unsubscribeFromTopics(token)                # 토큰 삭제 시 토픽 구독 해제
├── deactivateToken(token)                      # is_active = false 처리
├── deactivateAllUserTokens(userId)             # 특정 유저의 모든 토큰 비활성화 (회원 탈퇴용)
├── cleanupStaleTokens()                        # 90일 미사용 토큰 일괄 삭제 (Cron용)
└── getActiveTokensByUserId(userId)             # 유저의 활성 토큰 배열 조회
```

**`subscribeToTopics()` 병원 토픽 구독 상세:**
```
subscribeToTopics(token, userId) {
  1. admin.messaging().subscribeToTopic([token], 'all_users')  // 전체 유저 토픽
  2. users 테이블에서 hospitalId 조회 (userId로 DB JOIN)
     - users.hospitalId가 있으면 → subscribeToTopic([token], `hospital_${hospitalId}`)
     - users.hospitalId가 null이면 → 병원 토픽 구독 생략
```

## 6. 기존 파일 수정 상세

### A. `server/services/notifications.ts` — FCM 연동 레이어 추가

**변경 내용:** `createNotification()` 함수 내부에서, DB 저장 성공 후 → FCM 발송을 자동 실행하도록 연동

```
현재:
  createNotification(data) → DB INSERT → return notification

변경 후:
  createNotification(data) → DB INSERT → notification_id 획득
    → shouldSendNotification() 검사
    → users.is_system_push_agreed 검사
    → push.service.sendToUser(userId, payload) 호출 (비동기, 발송 실패 시에도 알림은 유지)
    → return notification
```

> `action_url`, `image_url` 매개변수를 `data`에 추가. 기존 호출부(`create*Notification` 6종)도 필요 시 action_url 전달하도록 수정.

**⚠️ 주의사항 1: `userId` 타입 변환 필수**

기존 `notifications` 테이블의 `userId`는 **`text` 타입** (schema.ts 1057줄: `userId: text("user_id")`), 하지만 `admin.mission.review.service.ts`에서 반환되는 `approved.userId`는 **`integer` 타입**입니다.

따라서 FCM 발송 연동 코드에서 반드시 **`String(userId)` 변환**을 적용해야 합니다:
```typescript
// ❌ 잘못된 호출 (타입 불일치로 DB INSERT 실패 가능)
createNotification({ userId: approved.userId, ... });

// ✅ 올바른 호출 (integer → string 변환)
createNotification({ userId: String(approved.userId), ... });
```

**⚠️ 주의사항 2: `shouldSendNotification()` switch-case에 미션 타입 추가 필수**

기존 `shouldSendNotification()` 함수의 switch-case에 `'mission_approved'`, `'mission_rejected'` 타입이 **없습니다**. 현재는 `default: return true`로 처리되어 당장 버그는 아니지만, 추후 카테고리별 ON/OFF 제어를 위해 **명시적 case를 추가**해야 합니다:

```typescript
// notifications.ts의 shouldSendNotification() switch문에 추가할 case
case 'mission_approved':
case 'mission_rejected':
  return settings.applicationStatusChanges ?? true;
```

#### 개별 승인/반려: `approveSubmission()` (309줄), `rejectSubmission()` (339줄)

```
현재:
```

#### ⚠️ 일괄 처리: `bulkUpdateStatus()` (370줄) — **이 함수도 반드시 FCM 트리거 추가**

`bulkUpdateStatus()`는 관리자가 체크박스로 여러 제출물을 한 번에 승인/반려할 때 호출됩니다. 이 함수에 FCM 트리거를 빠뜨리면 **일괄 승인 시 푸시가 나가지 않는 버그**가 발생합니다.

```
변경 후:
  bulkUpdateStatus → DB UPDATE (inArray) → recalculateBigMissionProgress (uniqueUserIds)
    → updatedRecords에서 status가 APPROVED 또는 REJECTED인 레코드 필터
    → 각 유저별로 createNotification() 호출 (for...of로 순회)
    → return
```

> 💡 **일괄 발송 시 주의:** 10명의 유저를 한 번에 승인하면 10번의 `createNotification()`이 호출됩니다. 각각 비동기로 FCM 발송이 실행되므로 성능 이슈는 없지만, `push_delivery_logs`에 개별 이력이 아닌 **통합 이력 1건으로 기록**하는 것이 관리 편의에 좋습니다.

### C-2. `server/routes/auth.ts` — 로그아웃 시 토큰 비활성화 추가

> 🚨 **치명적 버그 방지:** 로그아웃 시 해당 기기의 FCM 토큰을 비활성화하지 않으면, 로그아웃한 유저에게 계속 푸시가 갔니다.
> 예: A유저가 로그아웃 → 같은 기기에서 B유저 로그인 → A의 미션 승인 알림이 B의 기기에 표시됨 (프라이버시 위반)

```
현재:

변경 후:
  POST /api/auth/logout
    → req.body.deviceToken이 있으면 → push.token.service.deactivateToken(deviceToken)
    → req.logout() → 세션 파기 → 쿠키 삭제 → return
```

**프론트엔드 연동 (클라이언트에서 로그아웃 호출 시):**

### D. `client/src/hooks/usePushNotifications.ts` — 인증 연동 + 딥링크 + 토스트

> 🚨 **치명적 버그 방지 — 로그인 전 토큰 증발 문제:**
> 현재 훅은 `App.tsx`에서 **무조건 호출**됩니다 (94줄). 로그인 여부와 무관하게 실행되므로, 앱 최초 설치 시(로그인 전) 푸시 권한 허용 → FCM 토큰 발급 → 서버 전송 → **401 실패**가 발생합니다.
> FCM 플러그인은 이미 발급한 토큰의 `registration` 이벤트를 **다시 쏘주지 않으므로**, 이후 로그인해도 토큰이 서버에 저장되지 않아 **영원히 푸시를 못 받는 유저**가 됩니다.

**해결 설계 — 인증 상태와 토큰 발송을 동기화:**

```typescript
export const usePushNotifications = (isAuthenticated: boolean) => {
  const [fcmToken, setFcmToken] = useState<string | null>(null);
  const [tokenSentToServer, setTokenSentToServer] = useState(false);

  // === Phase 1: FCM 토큰 발급 (로그인 여부와 무관하게 실행) ===
  useEffect(() => {
    if (!Capacitor.isNativePlatform()) return;

    const init = async () => {
      const permission = await PushNotifications.requestPermissions();
      if (permission.receive === 'granted') {
        await PushNotifications.register();
      }
    };
    init();

    // registration 이벤트: 토큰을 React State에만 저장 (서버 전송 안 함!)
    PushNotifications.addListener('registration', (token: Token) => {
      setFcmToken(token.value);  // ✅ State에 보관만
      // ❌ 여기서 fetch('/api/users/device-token') 호출 금지!
      // ❌ 로그인 전이면 401 실패하고, 이 이벤트는 다시 안 뜨므로 토큰 증발
    });

    // ... 딥링크, 토스트 리스너는 아래 코드와 동일 ...
  }, []);

  // === Phase 2: 로그인 되면 토큰을 서버에 전송 ===
  useEffect(() => {
    // 조건: (1) 로그인 상태이고, (2) 토큰이 있고, (3) 아직 서버에 안 보냈을 때
    if (isAuthenticated && fcmToken && !tokenSentToServer) {
      const sendTokenToServer = async () => {
        try {
          await fetch('/api/users/device-token', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ token: fcmToken, deviceType: Capacitor.getPlatform() })
          });
          setTokenSentToServer(true);  // 성공 시 중복 방지 플래그
        } catch (error) {
          console.error('Failed to send token:', error);
          // 500ms 후 1회 재시도
        }
      };
      sendTokenToServer();
    }
  }, [isAuthenticated, fcmToken, tokenSentToServer]);

  return { fcmToken };
};
```

**핵심 변경점 요약:**

| 기존 | 변경 후 |
|------|--------|
| `registration` 리스너 안에서 즉시 `fetch()` | `registration`에서는 **State에만 저장** |
| 로그인 여부 무관 | `isAuthenticated`가 `true`일 때만 서버 전송 |
| 훅 매개변수 없음 | `isAuthenticated: boolean` 매개변수 추가 |
| `App.tsx`에서 `usePushNotifications()` | `usePushNotifications(!!user)` 로 변경 |

**추가 변경 사항 — 딥링크 + 토스트 (기존 계획과 동일):**

### A. 발송 화면

- **타겟팅 옵션:** 전체 발송 / 병원별 발송 / 개별 회원 검색 발송
- **2단계 확인 모달:** 전체 발송 시 `총 N명에게 발송합니다. 진행하시겠습니까?` 확인 다이얼로그

- **발송 가능 기기 수 표시:** 발송 대상 선택 시 `현재 푸시 수신 가능 기기: N대` 안내 문구 표시

> 💡 **왜 기기 수를 보여줘야 하나?** 토픽 구독자 수 ≠ 전체 앱 유저 수입니다. 앱 미설치 유저, 푸시 거부 유저, 토큰 미등록 유저는 토픽에 구독되지 않으므로, 관리자가 "전체 발송"을 눌러도 실제 수신 인원이 전체 회원 수보다 적을 수 있습니다. 이 숫자를 UI에 보여주면 운영자의 혼란을 방지합니다.

## 9. 클라이언트 딥링크 & 포그라운드 & 뱃지 설계

### A. 딥링크 처리 흐름

```
[앱 백그라운드/종료 상태]
사용자가 푸시 알림 터치
  → pushNotificationActionPerformed 리스너 발동
  → notification.data.action_url 추출
  → React Router navigate(action_url)
  → 해당 화면 이동 + 알림 읽음 처리 API 호출

[앱 완전 종료(Terminated) 상태 — 콜드 스타트 방어]
  → App.tsx 마운트 시 PushNotifications.getDeliveredNotifications() 호출
  → 미처리 알림이 있으면 action_url로 이동
```

### B. 포그라운드 알림 표시 (이중 알림 방어)

```
앱이 켜져 있는 상태에서 푸시 수신
  → pushNotificationReceived 리스너 발동
  → if (platform === 'ios') → 아무것도 안 함 (OS가 네이티브 배너 자동 표시)
  → if (platform === 'android') → use-toast로 인앱 토스트 표시
```

> ⚠️ **전제 조건:** 위 이중 알림 방어 로직은 반드시 `capacitor.config.ts`에 `presentationOptions: ["badge", "sound", "alert"]` 설정이 **먼저 적용된 상태**에서만 의미가 있습니다. 이 설정이 없으면 iOS 포그라운드에서 OS 배너가 뜨지 않아, toast도 생략하면 **아무런 알림이 표시되지 않는 상황**이 됩니다.
> → 따라서 **Step 4** (클라이언트 확장)에서 `capacitor.config.ts` 수정과 포그라운드 토스트 로직을 **반드시 함께 적용**해야 합니다.

### C. 알림 뱃지 초기화

- 사용자가 🔔 알림장 페이지 진입 시 → `Badge.clear()` 호출 + 일괄 읽음 처리

## 10. 보안 및 운영 설계

### A. Rate Limiting (API 남용 방지)

| 엔드포인트 | 제한 | 이유 |
|-----------|------|------|
| `POST /api/admin/push/send` (전체) | 분당 1회 | 전체 발송 연타 방지 |
| `POST /api/admin/push/send` (개별/병원) | 분당 5회 | 합리적 수동 발송 |
| `POST /api/users/device-token` | 분당 10회/유저 | 토큰 갱신 남용 방지 |

### B. 좀비 토큰 에러 핸들링

| FCM 에러 코드 | 원인 | 처리 |
|--------------|------|------|
| `UNREGISTERED` | 앱 삭제 / 토큰 만료 | 즉시 토큰 DELETE + 토픽 구독 해제 |
| `SENDER_ID_MISMATCH` | FCM 프로젝트 불일치 | 즉시 토큰 DELETE |
| `INVALID_ARGUMENT` | Payload 오류 | 로그만 기록 (토큰 삭제 금지) |
| `QUOTA_EXCEEDED` | FCM 일일 할당량 초과 | 로그 + 관리자 알림 |

### C. 토큰 정기 정리 (Cron)

- 구현 방법: `node-cron` 라이브러리로 서버 프로세스 내에서 실행
- 주기: 매일 새벽 3시
- 대상: `last_used_at`이 90일 이상 경과한 `is_active=true` 토큰 → `is_active=false` 처리 후 30일 뒤 DELETE
- 90일 이상 지난 알림 (`notifications.created_at`) → 영구 삭제 (DB 용량 최적화)

### D. 프로덕션 로깅 정책

- 토큰 값은 프로덕션에서 절대 전체 로깅 금지 → 앞 10자만 출력
- 발송 결과는 `push_delivery_logs` 테이블에 저장
- 에러는 `winston` 로거의 `error` 레벨로 기록
