# 모놀리식 파일 아키텍처 분리 계획서 (PDCA - Plan)

> **작성일**: 2026-03-04
> **대상 지점**: `server/routes/mission-routes.ts` (5,132줄) 등 백엔드/프론트엔드 내 거대 파일 5종

## 1. 개요 및 배경 (Background)

현재 프로젝트 내에 단일 코드 라인이 3000~5000줄에 육박하는 거대 파일들이 다수 식별되었습니다. 이를 "모놀리식 안티패턴(God Object)"이라고 부르며, 상태의 복잡도를 증가시켜 심각한 유지보수 비용과 에러율 상승(Side effects)을 유발합니다. 향후 **10만 유저 수용 및 안정성 확보**를 위해 핵심 비즈니스 로직에 대한 **아키텍처 레이어링(Layering)** 및 코드 분할 리팩토링이 필수적입니다.

## 2. 분리 전략 및 아키텍처 방향성 (Architecture Strategy)

### [전략 1] 백엔드 - Controller/Service 패턴 적극 도입

가장 심각한 백엔드 라우터(예: `mission-routes.ts`)부터 `Routes - Controllers - Services` 계층으로 분리합니다.

1. **Routes Layer**: URL 매핑과 미들웨어(인증 등) 처리만 담당.
2. **Controller Layer**: 클라이언트 요청 파라 파라미터(Req/Res) 검증 및 처리. 비즈니스 로직은 최소화.
3. **Service Layer**: 순수 비즈니스 로직(DB 쿼리, 트랜잭션, 타 모듈 연동) 집약. 도메인별 분리(예: `MissionParticipationService`, `MissionRewardService`).

### [전략 2] 프론트엔드 - 뷰(View)와 상태/로직(Hook)의 분리

크기가 3600줄에 육박하는 `MissionManagement.tsx`와 1440줄의 `ImageGenerationTemplate.tsx`를 대상으로 합니다.

1. **Custom Hook 분리**: 데이터 페칭(TanStack Query), 상태 관리 로직을 모두 분리하여 `useImageGeneration.ts` 등의 훅으로 뽑아냅니다.
2. **Sub-Components 분할**: 거대한 파일 내부에 존재하는 모달 그리드, 폼 컴포넌트들을 각각의 기능별 독립 파일로 분리합니다 (예: `components/admin/mission/MissionGrid.tsx`).

## 3. 리팩토링 추진 단계 (Phasing)

이 리팩토링은 한 번에 모든 것을 뜯어 고치는 빅뱅(Big Bang) 방식이 아닌, **가장 크고 위험한 파일부터 차례대로** 쪼개는 점진적 스텝으로 진행합니다.

### Phase 1: 가장 위험한 バックエンド (Backend) 분리

- **대상**: `server/routes/mission-routes.ts` (5132줄)
- **작업 내용**:
  - `mission-routes.ts`를 세 개의 서비스로 분리.
  - 1. 미션 조회 및 기본 CRUD -> `mission.service.ts`
  - 2. 서브미션 및 참여 로직 -> `submission.service.ts`
  - 3. 리뷰 및 피드백 처리 -> `review.service.ts`
- **효과**: 라우터 코드를 1000줄 이하로 대폭 감소시켜, 트랜잭션 및 비즈니스 로직 추적이 용이해짐.

### Phase 2: 핵심 프론트엔드 컴포넌트 분할

- **대상**: `client/src/components/admin/MissionManagement.tsx` (3611줄)
- **작업 내용**:
  - 상단 필터/검색 영역 -> `MissionFilterBar.tsx` 분리
  - 본문 미션 목록 리스트 -> `MissionTable.tsx` 분리
  - 상태 조작(Hooks) 분리 -> `useMissionManagement.ts` 생성
- **효과**: UI 디자인 수정 시 5000줄을 스크롤하며 버그를 찾는 피로도 감소.

### Phase 3: AI 이미지 생성 템플릿 정규화

- **대상**: `client/src/components/ImageGenerationTemplate.tsx` (1440줄)
- **작업 내용**:
  - Firebase Storage 업로드 로직과 AI 생성 API 호출을 `useImageGeneration` 커스텀 훅으로 추출.
  - UI 컴포넌트의 다크/라이트 모드 테마 클래스 전면 전환 (앞서 분석한 UI 불일치 문제 동시 해결).

리팩토링 과정에서 발생하는 "로직 단절" 및 "의존성 파괴"를 방지하기 위해 다음과 같이 검증합니다.

1. **Gap Analysis (/check 워크플로우 대용)**: 분리 이전과 이후의 엔드포인트 응답(Response Schema)이 정확히 100% 일치하는지 `curl` 혹은 Postman(수동)으로 비교.
2. **TypeScript Checker**: TS 컴파일 에러(`npm run build`)가 0(Zero)인지 확인하여 타입 단절 방지.
